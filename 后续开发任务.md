## 最新人工快评观察（2026-05-18）

已回收 3 份朋友快评表和少量聊天反馈，共 13 条有效项目评分。样本量仍小，且有 2 份评分偏宽松，因此当前不能把平均分当成质量结论；更应该把低分项和文字反馈作为下一轮迭代方向。

关键发现：

- 朋友最关注的不是“项目本身好不好”，而是报告是否能帮助他更快判断“值不值得点开”。
- 最影响点开的字段是“推荐理由”和“为什么值得关注”；聊天反馈中明确提到“主要看推荐理由”“最想看到的是为什么值得关注”。
- 评分维度里最低的是“风险识别”，其次是“阅读路径”。典型反馈包括 API 不稳定、数据安全和隐私风险、项目庞大、学习曲线陡、第三方 API 依赖多。
- 当前评测说明仍容易让朋友误解成“给 GitHub 项目评分”，下一轮必须明确评分对象是“RepoPulse 的推荐判断是否可信”。
- 报告内容需要从“字段完整”转向“帮助点开”：先给是否值得点开的判断，再给值得关注、推荐理由、关键风险，事实来源和长阅读路径后置。

下一轮内容方向：

1. 单项目卡片前 30 秒要回答：现在为什么值得关注、为什么适合我、最大风险是什么。
2. “推荐阅读路径”从长步骤改成短行动建议，例如“先看 README 哪一节”“clone 后让 AI 分析哪些文件/模块”。
3. 风险必须更具体，优先覆盖 API 兼容、数据安全/隐私、维护节奏、license、依赖外部服务、项目过大或过早期。
4. 事实来源继续保留，但作为可信证据脚注，不抢占首屏判断空间。
5. 人工评测表从 6 个细分分数简化为更贴近产品价值的问题：想不想点开、最影响判断的字段、最大问题、最啰嗦的地方。
6. 推荐必须继续考虑用户画像，但画像不能只停留在语言、topic 和学习目标；后续要补充用户技术水平、已掌握技术栈、薄弱项、可投入时间和项目规模偏好，并估计当前项目对该用户的学习成本。

  - learning_score_reasonableness
  - reading_path_actionability
  - risk_quality
  - profile_fit_quality
- 每个项目新增轻量产品问题：

  - `decision_driver`：最影响判断的字段，枚举为为什么值得关注 / 推荐理由 / 风险 / 学习价值 / 事实来源 / 评分。
  - `biggest_problem`：最大问题，一句话。
  - `most_verbose_part`：最啰嗦或最想缩短的部分，一句话。

  - 最常见的 decision_driver 分布
  - 高频问题词：API 不稳定、隐私、license、学习曲线、项目过大、外部依赖

- 将评测结果写入 README 或 docs，用作项目可信度展示。

验收标准：

- 至少 20 个样本完成评分。
- 至少 5 人完成轻量快评，能说清楚“哪些字段最影响用户点开”。
- 能说清楚 RepoPulse 当前推荐质量的数字基线和定性问题 Top 3。
- 发现的问题能回流到 prompt、评分或采集规则。

## P2：优化日报卡片的信息顺序和风险质量

当前痛点：报告字段已经完整，但朋友反馈显示真正驱动点开的字段集中在“推荐理由”“为什么值得关注”和“风险”。如果继续平均展示所有字段，会显得啰嗦，也会稀释最有价值的判断。

- 调整单项目卡片顺序：
  1. 是否值得点开：一句话结论。
  2. 为什么值得关注：趋势信号、社区信号、近期更新。
  3. 推荐理由：结合当前画像说明具体价值。
  4. 最大风险：最多 3 条，必须具体。
  5. 推荐动作：先看 README 哪一节，或 clone 后让 AI 分析哪些入口/模块。
  6. 事实来源：GitHub metadata / README / release / topics / license，作为证据脚注后置。
- 压缩“学习价值明细”和长阅读路径：
  - 评分明细默认摘要展示，完整明细可保留在结构化数据或详情页。
  - 阅读路径从 5 步长清单改为 1 到 3 个下一步动作。
  - 增加“交给 AI 分析”的行动建议：入口文件、核心模块、可复刻功能、主要风险。
- 增强风险生成：
  - API 兼容和破坏性变更。
  - 数据安全、隐私、token/API key 处理。
  - license 缺失或商业使用不明。
  - 依赖外部服务、第三方 API、账号授权和平台政策。
  - 项目庞大导致学习成本高。
  - 项目过早期、维护节奏不明或 release 不稳定。
- 增加“推荐过强”检查：
  - 如果强推荐项目存在明显高风险，降级为值得观察。
  - 如果用户画像匹配弱但热度高，进入“上升很快，值得观察”而不是深读。
- 在卡片中预留“学习成本”位置，但先用规则/AI 估算，不要求第一版非常精细：
  - 低：30 分钟内能判断价值，半天内能跑通 demo。
  - 中：需要补 1 到 2 个前置知识，适合周末深入。
  - 高：依赖复杂环境、大型源码或陌生技术栈，建议观察或拆成小目标。

验收标准：

- 朋友快评中“风险识别”均分提升到 4.2 以上。
- “最啰嗦的部分”反馈不再集中在学习路径或事实来源。
- 至少 70% 评审能指出报告中最影响自己点开的字段。
- Top 3 项目卡片在 30 秒内能读出是否值得点开、为什么、最大风险。

## P3：引入用户技术画像和学习成本估计（规划中，当前文档超前于代码）

当前痛点：系统已经把语言、topic、学习目标纳入画像匹配，但还没有真正理解用户的技术水平和可投入时间。同一个项目对不同用户的学习成本差异很大：对熟悉 Docker/TypeScript/Agent 的用户可能适合深读，对初学者可能只能作为趋势观察。

状态说明：`docs/05-AI分析与提示词契约.md` 和 `docs/07-日报内容与个性化规范.md` 已经写出 `learning_cost` 的目标形态，但实现尚未完成。P3 的第一步不是继续扩写文档，而是补齐代码链路，或把相关文档明确标注为“规划字段”。

- 扩展用户画像字段：
  - `skill_level`：junior / intermediate / senior。
  - `known_stack`：已掌握技术栈，例如 TypeScript、Python、React、Node.js、Docker、RAG。
  - `weak_areas`：薄弱或暂不想投入的方向，例如 Kubernetes、Rust、分布式系统。
  - `time_budget`：quick-scan / weekend / deep-study。
  - `preferred_project_size`：small / medium / large。
  - `goal_priority`：resume_project / learn_architecture / ship_demo / follow_trend。
- 在 AI 分析 schema 中新增 `learning_cost`：
  - `level`：low / medium / high。
  - `score`：0 到 100，分数越高表示越适合当前用户投入。
  - `estimated_effort`：例如“30 分钟了解 / 半天跑通 / 2-3 天深入”。
  - `prerequisites`：需要补齐的前置知识。
  - `blockers`：可能卡住用户的环境、依赖、源码规模或概念门槛。
  - `why_for_this_user`：结合当前画像解释为什么成本低/中/高。
- 在排序中引入学习成本影响：
  - 高学习价值 + 低/中学习成本：优先进入深读。
  - 高热度 + 高学习成本：默认进入“上升很快，值得观察”。
  - 高学习成本但简历/复刻价值高：进入“可转化为项目灵感”，并标注适合周末深挖。
  - 明显超出用户技术栈且不符合当前目标：降级或进入谨慎关注。
- 在报告中展示“学习成本”：
  - 估计投入时间。
  - 关键前置知识。
  - 推荐动作，例如“先看 demo，不建议直接读全量源码”。
  - 对当前用户的成本解释。
- 不把学习成本简单等同于风险：
  - 风险是项目自身的不确定性。
  - 学习成本是用户与项目之间的匹配成本。

验收标准：

- 同一批候选项目在不同 `skill_level` / `time_budget` 画像下，Top 10 排序有可解释差异。
- 报告能明确说明“为什么这个项目对当前用户偏难或刚好合适”。
- 人工快评中新增“学习成本估计是否合理”，70% 以上项目达到 4 分及以上。
- 高学习成本项目不会被轻易标为“强推荐”，除非同时给出明确投入建议和收益解释。
