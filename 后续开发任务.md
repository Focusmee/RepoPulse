## 最新人工快评观察（2026-05-18）

已回收 3 份朋友快评表和少量聊天反馈，共 13 条有效项目评分。样本量仍小，且有 2 份评分偏宽松，因此当前不能把平均分当成质量结论；更应该把低分项和文字反馈作为下一轮迭代方向。

关键发现：

- 朋友最关注的不是“项目本身好不好”，而是报告是否能帮助他更快判断“值不值得点开”。

- 当前评测说明仍容易让朋友误解成“给 GitHub 项目评分”，下一轮必须明确评分对象是“RepoPulse 的推荐判断是否可信”。
- 报告内容需要从“字段完整”转向“帮助点开”：先给是否值得点开的判断，再给值得关注、推荐理由、关键风险，事实来源和长阅读路径后置。

下一轮内容方向：

1. 单项目卡片前 30 秒要回答：现在为什么值得关注、为什么适合我、最大风险是什么。
2. “推荐阅读路径”从长步骤改成短行动建议，例如“先看 README 哪一节”“clone 后让 AI 分析哪些文件/模块”。
3. 风险必须更具体，优先覆盖 API 兼容、数据安全/隐私、维护节奏、license、依赖外部服务、项目过大或过早期。
4. 事实来源继续保留，但作为可信证据脚注，不抢占首屏判断空间。
5. 人工评测表从 6 个细分分数简化为更贴近产品价值的问题：想不想点开、最影响判断的字段、最大问题、最啰嗦的地方。
6. 推荐必须继续考虑用户画像，但画像不能只停留在语言、topic 和学习目标；后续要补充用户技术水平、已掌握技术栈、薄弱项、可投入时间和项目规模偏好，并估计当前项目对该用户的学习成本。

## P2：优化日报卡片的信息顺序和风险质量（已完成）

当前痛点：报告字段已经完整，但朋友反馈显示真正驱动点开的字段集中在“推荐理由”“为什么值得关注”和“风险”。如果继续平均展示所有字段，会显得啰嗦，也会稀释最有价值的判断。

- 调整单项目卡片顺序：
  1. 是否值得点开：一句话结论。
  2. 为什么值得关注：趋势信号、社区信号、近期更新。
  3. 推荐理由：结合当前画像说明具体价值。
  4. 最大风险：最多 3 条，必须具体。
  5. 推荐动作：先看 README 哪一节，或 clone 后让 AI 分析哪些入口/模块。
  6. 事实来源：GitHub metadata / README / release / topics / license，作为证据脚注后置。
- 压缩“学习价值明细”和长阅读路径：
  - 评分明细默认摘要展示，完整明细可保留在结构化数据或详情页。
  - 阅读路径从 5 步长清单改为 1 到 3 个下一步动作。
  - 增加“交给 AI 分析”的行动建议：入口文件、核心模块、可复刻功能、主要风险。
- 增强风险生成：
  - API 兼容和破坏性变更。
  - 数据安全、隐私、token/API key 处理。
  - license 缺失或商业使用不明。
  - 依赖外部服务、第三方 API、账号授权和平台政策。
  - 项目庞大导致学习成本高。
  - 项目过早期、维护节奏不明或 release 不稳定。
- 增加“推荐过强”检查：
  - 如果强推荐项目存在明显高风险，降级为值得观察。
  - 如果用户画像匹配弱但热度高，进入“上升很快，值得观察”而不是深读。
- 在卡片中预留“学习成本”位置，但先用规则/AI 估算，不要求第一版非常精细：
  - 低：30 分钟内能判断价值，半天内能跑通 demo。
  - 中：需要补 1 到 2 个前置知识，适合周末深入。
  - 高：依赖复杂环境、大型源码或陌生技术栈，建议观察或拆成小目标。

验收标准：

- 朋友快评中“风险识别”均分提升到 4.2 以上。
- “最啰嗦的部分”反馈不再集中在学习路径或事实来源。
- 至少 70% 评审能指出报告中最影响自己点开的字段。

## P3：引入用户技术画像和学习成本估计（已完成）

当前痛点：系统已经把语言、topic、学习目标纳入画像匹配，但还没有真正理解用户的技术水平和可投入时间。同一个项目对不同用户的学习成本差异很大：对熟悉 Docker/TypeScript/Agent 的用户可能适合深读，对初学者可能只能作为趋势观察。

状态说明：`learning_cost` 已从目标设计进入当前实现契约。后续如果继续调整字段语义，必须同步 schema、prompt、heuristic、ranker、report、缓存版本和测试。

- 实施约束：
  - 先扩展并测试 `normalizeProfile`，所有新增字段必须有默认值和稳定类型，避免 prompt、ranker、report 读取到 `undefined`。
  - 不把学习成本只放在 AI 输出里；必须抽出统一的 `learningCost` 估算模块，heuristic、AI 降级、report 和 ranker 共用同一套语义。
  - AI 输出可以补充或覆盖学习成本解释，但 no-ai 和 AI 失败降级时也必须稳定生成 `learning_cost`。
  - P3 修改分析 schema 后必须升级 `ANALYSIS_CACHE_VERSION`，避免旧缓存缺字段却被继续复用。
  - 学习成本和风险保持边界：风险描述项目自身不确定性；学习成本描述当前用户投入门槛。
- 扩展用户画像字段，并在 `src/config/profile.js` 归一化：
  - `skill_level`：junior / intermediate / senior。
  - `known_stack`：已掌握技术栈，例如 TypeScript、Python、React、Node.js、Docker、RAG。
  - `weak_areas`：薄弱或暂不想投入的方向，例如 Kubernetes、Rust、分布式系统。
  - `time_budget`：quick-scan / weekend / deep-study。
  - `preferred_project_size`：small / medium / large。
  - `goal_priority`：resume_project / learn_architecture / ship_demo / follow_trend。
- 在 AI 分析 schema 中新增 `learning_cost`：
  - `level`：low / medium / high。
  - `investment_fit_score`：0 到 100，分数越高表示越适合当前用户现在投入；避免和“成本高低”语义写反。
  - `estimated_effort`：例如“30 分钟了解 / 半天跑通 / 2-3 天深入”。
  - `prerequisites`：需要补齐的前置知识。
  - `blockers`：可能卡住用户的环境、依赖、源码规模或概念门槛。
  - `why_for_this_user`：结合当前画像解释为什么成本低/中/高。
- 在排序中引入学习成本影响：
  - 高学习价值 + 低/中学习成本：优先进入深读。
  - 高热度 + 高学习成本：默认进入“上升很快，值得观察”。
  - 高学习成本但简历/复刻价值高：进入“可转化为项目灵感”，并标注适合周末深挖。
  - 明显超出用户技术栈且不符合当前目标：降级或进入谨慎关注。
- 在报告中展示“学习成本”：
  - 估计投入时间。
  - 关键前置知识。
  - 推荐动作，例如“先看 demo，不建议直接读全量源码”。
  - 对当前用户的成本解释。
- 不把学习成本简单等同于风险：
  - 风险是项目自身的不确定性。
  - 学习成本是用户与项目之间的匹配成本。

验收标准：

- 同一批候选项目在不同 `skill_level` / `time_budget` 画像下，Top 10 排序有可解释差异。
- 报告能明确说明“为什么这个项目对当前用户偏难或刚好合适”。
- 人工快评中新增“学习成本估计是否合理”，70% 以上项目达到 4 分及以上。
- 高学习成本项目不会被轻易标为“强推荐”，除非同时给出明确投入建议和收益解释。
- `normalizeProfile` 能为旧画像配置补齐 P3 新字段默认值，并覆盖无效枚举值。
- no-ai、AI 成功、AI parse 失败降级三条路径都能生成结构一致的 `learning_cost`。
- 修改 P3 schema 后旧分析缓存不会被误复用。

## P4：继续增强评分和分层系统

当前能力：已有趋势分、学习价值分、画像匹配、项目类型识别和可信分层。

- 把仓库类型识别从关键词规则升级为更透明的多信号分类：
  - repo name
  - description
  - topics
  - README 章节
  - 文件目录结构
- 增加源码目录结构分析：
  - 是否有 `src/`
  - 是否有 `examples/`
  - 是否有 `docs/`
  - 是否有测试目录
  - 是否有 package/workspace/monorepo 信号
- 增加风险规则：
  - license 缺失
  - 最近维护弱
  - issue 过多且无 release
  - README 很短
  - 资源合集伪装成工程项目

验收标准：

- 资源合集、模板、示例集合误判率下降。
- “今日最值得深读”保持少而精。
- 分层原因可以被用户复核。

## P4.5：报告模型分层、中文化与场景解释

当前痛点：当前报告已经有推荐判断，但展示层仍会暴露 `low` / `medium` / `high` 等英文枚举；同时报告渲染和精华版生成仍存在从 Markdown 反向抽取的过渡方案。对于小红书、抖音、私信试读和低技术背景用户，还需要更快解释“这个项目是什么、在什么场景用、能学到什么”。

- 报告架构重构方向：
  - 新增报告模型构建层：从 `ranked.items`、`stats`、`profile`、`date` 生成标准 full report model。
  - 新增变体模型：从 full report model 提炼 brief、email、social card 等 view model。
  - 新增多格式渲染层：Markdown、HTML、Email HTML、PDF 都从 report model 渲染。
  - 当前 `extractBriefFromMarkdown` 保留为历史 Markdown 兼容适配器，不继续扩大业务逻辑。
- 展示层中文化：
  - `learning_cost.level` 展示为低成本 / 中等成本 / 高成本。
  - `risk.severity` 展示为低风险 / 中风险 / 高风险。
  - `learning_value.level` 展示为低 / 中 / 高。
  - 质量警告等级展示为中文，但保留英文 type 方便调试。
  - 内部 schema、排序和校验仍保留 `low` / `medium` / `high`。
- 增加项目背景与使用场景：
  - Top 项目新增“项目背景与使用场景”，解释项目是什么、解决什么问题、谁会用。
  - 新增“简单例子”，用一个具体场景说明项目发挥作用。
  - 新增“能学习到什么 / 可复刻点”，强调工程能力、技术模式和 demo 方向。
  - 控制长度：背景与场景 120 到 180 字；简单例子 1 段；能学习到什么 1 到 4 条。
- AI 契约扩展：
  - 后续 schema 可新增 `context_explanation`、`use_case_example`、`learning_takeaways`。
  - no-ai 和 heuristic 路径要有兜底生成策略。
  - 修改 schema 后必须升级 `ANALYSIS_CACHE_VERSION`。

验收标准：

- 完整版和精华版报告不再直接暴露英文等级作为主要用户文案。
- Top 3 项目能让不熟悉技术的读者在 30 秒内说清“它是什么、做什么、什么时候用”。
- 精华版和完整版共享同一套报告模型，不再依赖 Markdown 反向解析作为主路径。
- 旧 Markdown 报告仍可通过兼容适配器生成精华版。

## P4.6：Top 项目源码轻扫规划

当前能力边界：基础日报读取 GitHub 元数据、README 和 latest release，不读取完整仓库源码。报告中基于源码结构的判断必须保持克制，不能暗示系统已完成源码级审计。

- 实施原则：
  - 不做全仓库 clone。
  - 不对所有候选项目读取源码。
  - 只对 Top 1 到 3 项目做轻量源码信号补充。
  - 首版作为深度版 / 付费增强卖点，不作为基础日报前置条件。
- 数据范围：
  - 根目录目录树。
  - `src/`、`packages/`、`examples/`、`docs/`、`tests/` 等一级目录摘要。
  - 关键清单文件：`package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml` 等。
  - 少量入口文件：`src/index.*`、`main.py`、`app.*`、examples 中的最小样例。
- 分析输出：
  - 项目入口在哪里。
  - 模块边界是否清晰。
  - 新手先读哪些文件。
  - 哪些部分适合复刻成 demo。
  - 源码级风险，例如入口复杂、目录混乱、测试缺失、构建链复杂。
- 工程约束：
  - 源码轻扫结果进入独立缓存，避免每日报告重复请求。
  - AI 输入必须标注哪些内容来自源码轻扫，哪些只来自 README。
  - 报告中要显示“源码轻扫：已启用 / 未启用”。

验收标准：

- Top 3 项目能展示更可靠的入口文件和源码阅读路径。
- 未启用源码轻扫时，报告不会声称已验证源码结构。
- 源码轻扫失败不影响基础日报生成。

## 变现距离评估与开发规划

当前 RepoPulse 已经具备较完整的日报生成引擎、个性化画像、AI/heuristic 降级、质量检查和自动化测试基线，但它现在仍更接近“可收费验证前的产品化日报引擎”，还不是成熟 SaaS 产品。

阶段判断：

- 距离“可收费验证”：约剩余 40% 工作量。主要缺口是公开 demo、稳定投递、订阅名单、付费说明、质量基线和真实用户留存反馈。
- 距离“成熟 SaaS 化产品”：约剩余 70%+ 工作量。主要缺口是账号体系、订阅自助管理、正式支付、Web 历史归档、团队能力、运营后台和规模化稳定性。

优先变现形态：

- 首选路径：面向 AI 应用开发者的付费日报订阅。
- 首批目标用户：AI 应用开发者、Agent/RAG/开发者工具关注者、希望持续发现可复刻项目机会的独立开发者。
- 变现 MVP：AI Builder Radar 付费日报 / 周报订阅。
- 首版交付：邮件优先，Markdown / HTML 报告作为核心产物；不先做复杂账号系统和 Web Dashboard。
- 支付策略：首版不接入支付 SDK，先用外部支付或人工收款验证需求；代码侧只维护订阅名单、投递状态和报告归档。
- 产品指标：付费转化、打开率、点击意愿、续费意愿、每期至少 1 个可行动项目。

### M0：付费验证准备

目标：把当前项目包装成陌生 AI 开发者能理解、愿意试订的产品形态。

- 准备公开 demo report：
  - 使用 AI Builder 画像生成一份高质量样例日报。
  - 输出 Markdown 和 HTML 两种展示形态。
  - 在 README 中展示日报片段、核心差异和适合订阅的人群。
- 增加精华版 HTML 生成工具：
  - 从完整 Markdown 日报抽取 Top 3 深读项目和最多 2 个项目灵感。
  - 用于小红书、抖音、社群、私信和订阅页试读材料。
  - HTML 内置打印样式，PDF 首版通过浏览器打印导出，不先引入自动 PDF 依赖。
  - 后续重构为 report model -> brief model -> HTML renderer，不长期依赖 Markdown 反向解析。
- 固化 AI Builder Radar 定位：
  - 明确它不是 GitHub Trending 搬运，而是“帮 AI 应用开发者判断哪些开源项目值得点开、学习或转化为应用机会”。
  - 报告首屏继续围绕“为什么值得关注、为什么适合我、最大风险是什么”组织。
- 建立质量基线：
  - 延续 20 个样本 + 5 人轻量快评。
  - 指标重点从“字段完整”转向“是否帮助用户做点击和学习决策”。
  - 汇总点击意愿、最大问题、最影响判断字段和可行动项目数量。
- 准备订阅说明：
  - 明确更新频率：每日 Radar 或每周精选。
  - 明确适用用户、样例内容、早鸟价格或试用方式。
  - 明确首版是人工运营 + 自动生成，不承诺团队 SaaS 能力。

验收标准：

- 陌生 AI 开发者 30 秒内能看懂 RepoPulse 为什么值得订阅。
- README 或发布页能展示一份可信 demo report 和质量指标。
- 至少 5 名目标用户愿意试读或留下反馈。

### M1：手动付费订阅 MVP

目标：用最小工程成本验证是否有人愿意持续为高质量项目情报付费。

- 增加定时生成规划：
  - 每日生成 AI Builder Radar。
  - 每周生成趋势总结或“本周最值得深挖项目”。
  - 保持生成和投递分离，避免报告生成成功但投递失败时无法复盘。
- 增加本地订阅名单设计：
  - 订阅人邮箱。
  - 绑定画像，例如 `ai-builder`。
  - 订阅频率：daily / weekly。
  - 订阅状态：trial / active / paused / canceled。
  - 标签：early-access、paid、friend-review、beta 等。
- 增加投递状态设计：
  - `generated`：报告已生成。
  - `sent`：已发送。
  - `failed`：投递失败，需要重试或人工处理。
  - `skipped`：因暂停、取消或无有效内容跳过。
- 邮件投递优先：
  - 首版可以先用 SMTP、Resend、Buttondown、Substack 或人工发送中的一种。
  - 飞书、Telegram 保持为后续渠道，不作为收费验证前置条件。
  - 投递内容优先使用精华版 HTML，完整 Markdown 作为归档或完整版链接。
- 运营闭环：
  - 每期记录打开反馈、点击意愿和最有价值项目。
  - 每周复盘留存和退订原因。
  - 把反馈回流到画像、风险识别、项目类型识别和报告模板。

验收标准：

- 连续 14 天稳定生成并投递 AI Builder Radar。
- 至少 5 个真实订阅 / 试用用户进入名单。
- 每期报告至少包含 1 个用户认为“值得点开或收藏”的项目。
- 能复盘每次投递的生成状态、投递状态和失败原因。

### M2：产品化订阅系统

目标：当手动订阅验证有效后，再把重复运营动作产品化。

- 增加轻量订阅管理入口：
  - 用户可查看当前订阅频率和画像。
  - 用户可暂停、恢复或取消订阅。
  - 管理员可查看订阅状态和投递失败列表。
- 增加历史报告归档：
  - 按日期、topic、画像和推荐分层检索历史日报。
  - 支持查看项目详情、评分原因、风险和学习成本。
  - Web 只消费已有结构化报告，不重写推荐逻辑。
- 增加画像编辑：
  - 允许用户调整技术栈、薄弱项、时间预算、项目规模偏好和目标优先级。
  - 调整后下一期报告使用新画像。
- 评估正式支付接入：
  - 根据首批付费用户地区和支付习惯，再选择 Stripe、Lemon Squeezy、国内支付或平台订阅工具。
  - 支付状态同步到订阅状态，但不让支付系统反向污染推荐逻辑。

验收标准：

- 用户可自助订阅、修改画像、查看历史报告。
- 管理员可查看订阅、投递和失败状态。
- 正式支付方案有明确候选和接入边界。
- Web / 邮件 / 周报都复用同一份结构化报告数据。

### 变现实施计划：推广、成交与续费

目标：让开发计划不只停留在功能建设，而是每一阶段都有对应的获客、成交和留存动作，用真实付费反馈决定下一步开发优先级。

核心卖点：

- 每天节省筛选 GitHub 项目的时间，不再刷一堆热榜噪声。
- 每期给出“为什么值得点开、适不适合你、最大风险、下一步怎么学”。
- 面向 AI 应用开发者，重点发现 Agent、RAG、开发者工具、自动化工作流和可复刻产品机会。
- 不是卖“项目列表”，而是卖“更快判断哪个项目值得投入时间”的决策辅助。

推广材料准备：

- 一份公开 demo report：必须能展示 3 个强推荐项目、风险判断、学习成本和推荐动作。
- 一页订阅说明：包含适合谁、不适合谁、更新频率、样例截图、早鸟价格、试读方式和退款/取消规则。
- 一段 30 秒介绍文案：用于私信、朋友圈、社群和 README。
- 三类内容素材：
  - 项目拆解：挑 1 个热门 AI 项目，展示 RepoPulse 如何判断值不值得点开。
  - 趋势观察：每周总结 Agent/RAG/开发者工具方向的上升项目。
  - 对比内容：说明 RepoPulse 和 GitHub Trending、普通 newsletter、手动收藏列表的区别。

获客渠道：

- 第一圈层：朋友、同学、同事、正在做 AI 应用或找项目灵感的开发者，目标是拿到最早 5 个试读用户。
- 技术社区：掘金、知乎、V2EX、即刻、X/Twitter、开发者微信群或 Discord/Telegram 社群，优先发布项目拆解和周报精选，不直接硬广。
- 开源入口：README、GitHub release、示例日报、issue/discussion，吸引对项目本身感兴趣的人订阅 Radar。
- 内容入口：把每周精选沉淀成文章，结尾只放一个明确 CTA：“想每天收到 AI Builder Radar，可申请试读”。

成交路径：

1. 先发样例，不先推销：给目标用户发送一份 demo report 或一段 Top 3 摘要，询问“这份报告能不能帮你少刷 30 分钟 GitHub”。
2. 收集痛点：确认对方更关心找项目灵感、学习路线、简历项目、产品机会，还是趋势雷达。
3. 给出试读：提供 3 到 7 天试读，每期只问两个问题：
   - 今天有没有至少 1 个项目值得点开？
   - 哪个字段最影响你的判断？
4. 试读结束成交：如果连续多期有价值，给早鸟订阅选项，例如月付、季度付或一次性 beta 名额。
5. 人工收款优先：首版通过外部支付或人工收款完成，不为支付 SDK 提前投入开发。
6. 成交后进入订阅名单：记录邮箱、画像、频率、状态、来源渠道、成交价格和关键需求。

定价实验：

- 免费试读：3 到 7 天，只验证内容价值，不承诺长期免费。
- 早鸟个人版：低价月付或季度付，换取持续反馈和产品共创。
- 深度版候选：如果用户愿意为“每周 3 个项目深度拆解 + 学习/复刻建议”付更高价格，再规划周报增强。
- 暂不设计团队版价格；只有当出现多个团队需求后，再进入 M2/M3 评估。

转化指标：

- 触达数：每周主动触达多少目标用户。
- 试读转化率：愿意留下邮箱或接收试读的人数 / 触达人数。
- 付费转化率：试读后付费人数 / 试读人数。
- 内容有效率：每期认为“至少 1 个项目值得点开”的用户比例。
- 留存信号：连续 2 周仍打开、回复或收藏项目的用户数。
- 取消原因：太泛、太长、不够新、项目不可行动、价格不匹配、频率太高。

开发与变现联动规则：

- 如果用户说“内容有用但太长”，优先优化报告首屏和摘要，不先做 Web。
- 如果用户说“愿意看但不会付费”，优先调整定位、目标用户和定价，不先加复杂功能。
- 如果用户愿意付费但要求稳定送达，优先做邮件投递、订阅名单和投递状态。
- 如果用户反复查历史项目，才提前做历史报告归档和搜索。
- 如果多个用户要求不同画像，才建设画像编辑入口。
- 如果出现团队采购或多人共享需求，才评估账号、权限和团队版。

每周成交复盘：

- 本周触达了多少人，来自哪些渠道。
- 有多少人愿意试读，有多少人愿意付费。
- 用户最常提到的购买理由是什么。
- 用户最大的不购买理由是什么。
- 哪一期报告最能促成回复或成交。
- 下周应该改内容、改渠道、改定价，还是改产品功能。

### 变现阶段的接口约束

- 新增订阅配置时，至少包含：邮箱、画像、频率、状态、标签。
- 新增投递记录时，至少包含：报告日期、画像、渠道、状态、失败原因、发送时间。
- 日报生成与投递必须分离：生成失败、投递失败和用户状态跳过要能分别复盘。
- 现有 Markdown / 结构化日报继续作为核心数据源，Web 和邮件不得重新实现推荐排序。
- 首版不引入复杂账号系统、团队权限和正式支付 SDK；这些只在 M2 之后评估。
