# 产品化与变现路线

## 决策摘要

RepoPulse 的产品化默认路线是：先做“网站入口 + SBTI 测试 + 个性化开源雷达 / 日报订阅”，不先做完整 SaaS。

当前阶段要验证的不是用户是否需要一个复杂后台，而是：

- SBTI 测试能否把陌生用户转化成可服务的画像。
- 个性化日报是否真的帮用户节省筛项目时间。
- 用户是否愿意为持续、高质量、可行动的开源项目雷达付费。

只有当订阅交付和复购信号稳定后，再把历史归档、检索、收藏、团队协作升级为 dashboard 或 SaaS。

## 最终产品形态

完整形态可以拆成五层：

1. **公开网站入口**：介绍 RepoPulse、展示样例日报、承接 SBTI 测试和订阅转化。
2. **SBTI AI 开发者测试**：用轻松问题生成开发者行为风格、细化标签和 RepoPulse profile。
3. **个性化开源雷达**：按用户画像生成每日或每周 AI 开源项目推荐，解释为什么值得看、怎么开始、风险是什么。
4. **行动型项目档案**：沉淀历史推荐、项目灵感、源码轻扫、作品集建议和产品化机会。
5. **Pro / Team 工作台**：为团队或专业用户提供趋势跟踪、项目对比、定制方向、团队共享和更深分析。

第一阶段的核心交付不是 dashboard，而是一份用户愿意持续打开的高质量雷达报告。

## 网站 vs SaaS 决策

当前选择：先做网站入口和订阅式交付，后做 SaaS。

原因：

- RepoPulse 的首要风险是推荐质量、解释可信度和付费意愿，不是后台功能完整度。
- 网站能承担获客、测试、样例展示和购买入口，开发成本更低。
- 邮件、Markdown、HTML 精华版可以先验证内容价值，不需要登录和数据库。
- 完整 SaaS 会提前引入账号、权限、计费、通知、数据隔离和 dashboard 维护成本。

升级到 SaaS 的触发条件：

- 连续 4 周稳定交付，核心用户仍愿意打开和反馈。
- 出现稳定付费用户，并且用户主动要求历史检索、收藏、项目对比或团队共享。
- 单靠邮件和静态 HTML 已无法承载交付和留存。
- Team / Pro 用户愿意为定制方向、团队共享或深度分析付费。

## 推荐用户路径

面向最终用户，推荐路径固定为：

1. 做 SBTI 测试，得到一个有传播感的 AI 开发者人格结果。
2. 系统生成 RepoPulse profile，明确行业、目标、时间预算、痛点和解释风格。
3. 用户运行或订阅个性化日报，看到按自己画像筛过的项目。
4. 每个项目给出推荐理由、风险、学习成本和下一步动作。
5. 长期沉淀为个人项目发现档案、作品集候选库或产品机会库。

## 推荐开发路径

面向实现，开发路径固定为：

- 本地 CLI 优先：继续用 `npm run sbti` 和 `npm run report` 验证 profile 到日报的闭环。
- 静态结果页其次：SBTI 可输出本地 HTML 或结构化 `personaResult`，供社交分享和试读使用。
- 网站入口只调用 SBTI profile generator：网站负责收集答案和展示结果，不采集 GitHub、不分析 README、不跑 ranker。
- GitHub collector、analyzer、ranker、reporter 继续独立：SBTI 只通过 profile JSON 与日报主流程连接。

## 最近 30 天行动计划

接下来 30 天只追一个目标：证明 SBTI 能把用户带到一份“我愿意继续看”的个性化开源雷达。

### 第 1 周：打磨可展示闭环

- 准备 3 个代表性 SBTI profile 和对应样例日报。
- 生成本地结果页或 HTML 精华版，作为试读材料。
- 检查每份样例是否清楚回答：为什么推荐、适合谁、风险是什么、下一步做什么。
- 修掉会影响理解的文案、字段和画像展示问题。

### 第 2 周：小范围试读

- 找 5 到 10 个目标用户做 SBTI 并读样例日报。
- 记录完成率、理解成本、点开意愿、订阅意愿和最困惑的字段。
- 不急着做新功能，先判断问题是测试不准、日报不准，还是商业表达不清。

### 第 3 周：轻量网站入口

- 做一个只服务获客和试读的入口页。
- 页面包含 SBTI 测试、结果展示、样例日报和订阅 / waitlist CTA。
- 不做登录、不做复杂 dashboard、不让网站直接运行 GitHub 采集和日报主流程。

### 第 4 周：订阅交付实验

- 用每周雷达验证持续交付，比日更更适合早期质量控制。
- 提供免费样例、试读和一个明确的个人订阅方案。
- 若出现高意愿用户，再提供专题深度报告或项目包作为加购。

30 天结束时只做一个判断：继续优化报告和订阅，还是进入轻量 dashboard。若没有稳定打开率和付费信号，不推进 SaaS 化。

## 变现路径

第一阶段采用轻量商业化，不先建设完整 SaaS 计费后台。

| 层级 | 交付物 | 目标 |
| --- | --- | --- |
| 免费入口 | SBTI 测试、人格结果、样例日报 | 获客、传播、解释产品价值 |
| 个人订阅 | 每日 / 每周个性化 AI 开源雷达 | 验证持续付费和打开率 |
| 深度报告 | 专题项目包、源码轻扫、作品集路线 | 提高客单价，验证深度需求 |
| 求职 / 作品集路线 | 可复刻项目清单、简历表达、学习路径 | 服务职业成长用户 |
| Pro / Team | 定制方向、趋势跟踪、团队共享 | 在付费信号明确后再 SaaS 化 |
| 赞助 / 广告 | 工具、课程、招聘、开发者服务赞助 | 后置，避免早期影响推荐可信度 |

变现顺序应先证明“用户愿意为判断付费”，再证明“用户愿意为工具后台付费”。

## 验证指标

早期重点观察：

- SBTI 完成率：用户是否愿意完成测试。
- profile 到日报转化率：测试后是否愿意生成或订阅报告。
- 样例日报打开率：用户是否能在短时间理解价值。
- 付费意愿：是否有人愿意为连续交付或深度报告付费。
- 留存反馈：连续 2 到 4 周后，用户是否仍认为推荐节省时间。
- dashboard 需求强度：用户是否主动要求历史归档、收藏、搜索、对比和团队共享。

如果 SBTI 完成率高但日报付费弱，优先优化样例日报和推荐解释；如果日报付费强但管理成本高，再推进 SaaS 化。

## 边界

本路线不改变 RepoPulse 的核心工程边界：

- SBTI 不负责 GitHub 采集。
- SBTI 不直接改 ranker。
- 网站入口不直接运行日报主流程。
- 付费交付不允许牺牲推荐可信度。
- 赞助内容必须与推荐结果显式区分。

详细模块边界见 `docs/06-系统架构与模块边界.md`，SBTI 模块手册见 `src/persona/README.md`。
