# 数据源与数据治理规范

## 数据治理目标

RepoPulse 的数据层要支持长期稳定运行、可复盘评分、可解释推荐。数据不要求一开始很全，但必须干净、可追踪、可降级。

## 数据源分层

### 第一层：候选发现源

| 来源 | 用途 | 风险 |
| --- | --- | --- |
| GitHub Trending | 捕捉短期爆发项目 | 官方无稳定公开 API，抓取需谨慎 |
| GitHub Repository Search | 按语言、topic、更新时间补充候选 | 受 API 限流、排序规则影响 |
| 用户 watchlist | 覆盖个人关心领域 | 需要手动维护 |

### 第二层：项目上下文源

| 来源 | 用途 |
| --- | --- |
| Repository metadata | star、fork、language、topics、license、pushed_at |
| README | AI 分析的核心依据 |
| Releases | 判断成熟度和维护节奏 |
| Issues / PR | 判断活跃度和社区反馈 |
| package / manifest 文件 | 后续用于识别技术栈 |

## 候选项目采集规则

### Trending 采集

采集维度建议包括：

- daily。
- weekly。
- 用户关注语言，例如 Java、TypeScript、Python。
- 通用榜单。

采集结果必须记录 `source = github_trending` 和具体维度，例如 `trending:daily:typescript`。

### Search 采集

搜索条件应来自用户画像和系统 watchlist：

- language。
- topic。
- keyword。
- created date。
- pushed date。
- stars range。

采集结果必须记录查询条件，方便后续解释某项目为什么进入候选池。

### Watchlist 采集

watchlist 用于补足个人偏好：

```yaml
topics:
  - agent
  - rag
  - distributed-systems
keywords:
  - "spring ai"
  - "workflow engine"
repos:
  - owner/name
```

## 数据去重规则

仓库唯一键使用 GitHub `repo_id`。如果暂时没有 `repo_id`，使用 `owner/name` 作为临时唯一键。

同一仓库来自多个来源时：

- 合并来源列表。
- 保留最新元数据。
- 在日报中可说明“来自 Trending + 用户关注 topic”。

## 核心数据实体

### `repos`

保存仓库相对稳定的信息：

| 字段 | 含义 |
| --- | --- |
| `repo_id` | GitHub 仓库 ID |
| `owner` | 仓库 owner |
| `name` | 仓库名 |
| `full_name` | owner/name |
| `description` | 描述 |
| `language` | 主语言 |
| `topics` | topics 列表 |
| `license` | license |
| `html_url` | GitHub 链接 |
| `created_at` | 创建时间 |
| `first_seen_at` | RepoPulse 首次发现时间 |

### `repo_snapshots`

保存每日变化：

| 字段 | 含义 |
| --- | --- |
| `repo_id` | 仓库 ID |
| `snapshot_date` | 快照日期 |
| `stars` | star 总数 |
| `forks` | fork 总数 |
| `open_issues` | open issue 数 |
| `watchers` | watchers |
| `pushed_at` | 最近 push 时间 |
| `source_count` | 当天来源数量 |

唯一约束：`repo_id + snapshot_date`。

### `repo_documents`

保存 AI 分析上下文：

| 字段 | 含义 |
| --- | --- |
| `repo_id` | 仓库 ID |
| `readme_text` | README 文本 |
| `readme_fetched_at` | README 抓取时间 |
| `latest_release_notes` | 最新 release notes |
| `document_hash` | 文档 hash，用于判断是否需要重新分析 |

### `analysis_results`

保存 AI 结构化分析和评分依据：

| 字段 | 含义 |
| --- | --- |
| `repo_id` | 仓库 ID |
| `analysis_date` | 分析日期 |
| `model` | 使用模型 |
| `input_hash` | 输入 hash |
| `structured_json` | AI 原始 JSON |
| `confidence` | AI 置信度 |
| `validation_status` | schema 校验结果 |

### `daily_reports`

保存日报输出：

| 字段 | 含义 |
| --- | --- |
| `report_date` | 日期 |
| `profile_id` | 用户画像 ID |
| `markdown` | Markdown 内容 |
| `items_json` | 结构化项目列表 |
| `generated_at` | 生成时间 |
| `delivery_status` | 投递状态 |

## 数据质量规则

每日任务结束后至少检查：

- 候选项目数是否达到阈值。
- 去重后项目数是否异常偏低。
- README 获取成功率。
- AI 分析成功率。
- 最终可排序项目数。
- 日报 Top 项目是否存在重复。

建议初始阈值：

| 指标 | 阈值 |
| --- | --- |
| 原始候选项目数 | >= 50 |
| 去重后候选项目数 | >= 30 |
| README 获取成功率 | >= 70% |
| AI 分析成功率 | >= 80% |
| 最终推荐项目数 | >= 10 |

## 限流与降级

GitHub API 和页面抓取都可能失败。系统必须支持：

- 请求重试。
- 指数退避。
- 读取 API rate limit 响应信息。
- 缓存 README 和元数据。
- 单项目失败不中断整批任务。
- 当前数据不足时使用已有快照降级。

## 数据保留策略

建议：

- `repos` 长期保留。
- `repo_snapshots` 至少保留 180 天。
- `repo_documents` 保留最近版本和 hash。
- `analysis_results` 至少保留 90 天。
- `daily_reports` 长期保留，作为个人学习档案。

## 隐私与安全

首版个人数据主要是用户画像，不涉及敏感隐私。仍需遵守：

- GitHub token 不进入日志。
- 邮件配置不进入仓库。
- 用户画像中的私人目标不公开推送。
- 日报归档公开前需要确认是否包含个人偏好。
