# 系统架构与模块边界

## 架构目标

RepoPulse 的架构应支持：

- 每日稳定运行。
- 模块职责清晰。
- 数据可复盘。
- AI 输出可替换。
- 后续可以从 Markdown 日报扩展到邮件和 Web。

## 主流水线

```text
Profile Loader
→ Candidate Collectors
→ Metadata Hydrator
→ Snapshot Store
→ Trend Scorer
→ Document Fetcher
→ AI Analyzer
→ Learning Scorer
→ Personalized Ranker
→ Report Renderer
→ Report Store / Notifier
```

## 模块边界

### `profiles`

职责：

- 读取用户画像。
- 校验画像字段。
- 提供偏好、排除项、日报数量等配置。

不做：

- 不直接采集 GitHub。
- 不决定最终排序。

### `collectors`

职责：

- 从 GitHub Trending、Search、watchlist 收集候选项目。
- 给候选项目打来源标签。

不做：

- 不计算最终分数。
- 不调用 AI。
- 不生成日报文案。

### `hydrators`

职责：

- 获取仓库元数据。
- 规范化 owner、name、repo_id、topics 等字段。

不做：

- 不判断项目是否值得学。

### `store`

职责：

- 保存 repos、snapshots、documents、analysis、reports。
- 提供幂等写入。

不做：

- 不包含复杂业务判断。

### `scorers`

职责：

- 计算 trend_score。
- 计算 learning_score。
- 输出分项分和理由。

不做：

- 不直接请求外部 API。
- 不渲染日报。

### `ai`

职责：

- 基于输入上下文生成结构化分析。
- 校验 AI 输出 schema。
- 记录模型、输入 hash、输出 JSON。

不做：

- 不作为事实来源。
- 不绕过 schema 输出自由文本。

### `rankers`

职责：

- 结合 trend_score、learning_score、profile_match_score 和 penalty 排序。
- 给出分层推荐结果。

不做：

- 不重新抓取数据。

### `reports`

职责：

- 将结构化推荐结果组织为稳定的报告模型。
- 从完整报告模型提炼精华版、邮件摘要或社交平台素材。
- 将报告模型渲染为 Markdown、HTML 或邮件；PDF 由 HTML 打印样式或后续导出器生成。
- 保持日报模板稳定。

不做：

- 不改变评分。
- 不调用采集器。
- 不从 Markdown 反向推断业务判断，除非是兼容历史报告的临时适配器。

### `report builders`

职责：

- 把 `ranked.items`、`stats`、`profile` 和 `date` 转成完整报告模型。
- 从完整报告模型中提炼精华版模型，例如 Top 3 深读项目和最多 2 个项目灵感。
- 统一处理展示层字段，例如中文化的风险等级、学习成本等级、项目背景、使用场景和简单例子。

不做：

- 不输出 Markdown / HTML / PDF。
- 不重新排序项目。
- 不请求 GitHub 或 AI。

### `report renderers`

职责：

- 接收报告模型并输出不同格式。
- 支持完整版 Markdown、精华版 HTML，后续可扩展完整版 HTML、邮件 HTML 和 PDF。
- 保持格式相关逻辑，例如 HTML 转义、移动端样式、打印样式和邮件兼容样式。

不做：

- 不直接依赖 `ranked.items` 的内部排序逻辑。
- 不修改报告模型内容。
- 不把 PDF 变成新的业务数据源；PDF 只应由 HTML 或报告模型派生。

### `jobs`

职责：

- 编排完整日报任务。
- 管理任务状态、错误、重试和日志。

不做：

- 不写复杂业务逻辑；业务逻辑应放回对应模块。

## 数据流约束

推荐数据流：

```text
raw candidates
→ normalized candidates
→ repo metadata
→ snapshots
→ scored candidates
→ AI analysis
→ ranked items
→ report model
→ report variant model
→ renderer
→ Markdown / HTML / Email / PDF
```

禁止反向依赖：

- report renderer 不应反过来影响 scorer。
- renderer 不应从 Markdown 中反向解析业务字段；当前 Markdown 抽取精华版只作为过渡适配器。
- brief / full / email / social card 应共享同一份报告模型，不应各自重复实现字段选择逻辑。
- AI analyzer 不应直接写日报。
- collector 不应知道日报模板。
- store 不应知道业务评分公式。

## 幂等要求

每日任务必须可以重复执行。

要求：

- `repo_id + snapshot_date` 唯一。
- `report_date + profile_id` 唯一。
- AI 分析可通过 `input_hash` 判断是否复用。
- 重跑时可以选择覆盖报告，但不能产生重复项目。

## 错误处理

错误分三级：

| 级别 | 示例 | 处理 |
| --- | --- | --- |
| 项目级 | 某个 README 获取失败 | 记录失败，跳过或降级 |
| 阶段级 | AI 批量失败率过高 | 生成部分日报并标记质量告警 |
| 任务级 | GitHub API 完全不可用 | 任务失败，保留日志和重试入口 |

## 观测指标

每日任务应记录：

- 候选项目数。
- 去重后项目数。
- 元数据获取成功率。
- README 获取成功率。
- AI 分析成功率。
- 平均 AI 成本。
- 最终推荐项目数。
- 日报生成耗时。
- 失败原因 Top N。

## 技术选择约束

首版技术栈可以保持轻量：

- 数据库：Supabase / PostgreSQL。
- 任务调度：本地 cron、GitHub Actions 或后续 serverless cron。
- AI：通过统一 adapter 封装，避免业务代码绑定具体模型。
- 日报：先稳定报告模型，再渲染 Markdown / HTML / email；PDF 优先由 HTML 打印样式导出。

任何引入新基础设施的决定，都需要在 `09-迭代路线图与决策记录.md` 记录 ADR。

## 扩展方向

可预留但不急着实现：

- Web dashboard。
- 向量相似项目检索。
- 周报 / 月报。
- 多用户系统。
- 反馈学习。
- 项目学习计划生成。
- Top 项目源码轻扫：只对 Top 1-3 项目读取目录树和少量关键入口文件，用于深度版判断。
