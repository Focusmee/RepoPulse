# 配置与推荐可信度提升指南

本文档回答 5 个问题：

1. 如何配置 `GITHUB_TOKEN`
2. 如何配置 OpenAI Key
3. 如何优化 AI 分析 prompt
4. 如何把学习价值评分拆成可解释明细
5. 如何做 20 个项目的人工评测样本

## 1. 配置 GITHUB_TOKEN

RepoPulse 不配置 `GITHUB_TOKEN` 也能跑，但 GitHub 匿名 API 限流很低，容易影响候选项目数量和 README 获取成功率。

建议做法：

1. 打开 GitHub 的 Personal access tokens 页面。
2. 创建一个只用于 RepoPulse 的 token。
3. 只需要读取公开仓库信息，不需要写权限。
4. 把 token 写入项目根目录 `.env`。

`.env` 示例：

```env
GITHUB_TOKEN=ghp_xxx
```

验证：

```bash
npm run doctor
```

看到：

```text
GITHUB_TOKEN：已设置
```

就说明 RepoPulse 已经能使用认证请求。

## 2. 配置 OpenAI Key

RepoPulse 当前支持两种分析模式：

- 未配置 OpenAI：使用本地启发式规则分析。
- 已配置 OpenAI：优先调用 OpenAI，失败时自动降级到本地规则。

`.env` 示例：

```env
OPENAI_API_KEY=sk-xxx
OPENAI_MODEL=你的模型名
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_TIMEOUT_MS=60000
```

说明：

- `OPENAI_API_KEY` 是你的 API key。
- `OPENAI_MODEL` 填你账户当前可用、适合结构化 JSON 输出的模型名。
- `OPENAI_BASE_URL` 默认是官方地址，通常不用改。
- `OPENAI_TIMEOUT_MS` 建议先设为 `60000`，OpenAI-compatible provider 在长 README 输入下更容易超过默认 30 秒。

### OpenAI-compatible / DeepSeek 配置

RepoPulse 通过 Chat Completions 接口调用模型，因此可以接 OpenAI-compatible provider，例如 DeepSeek：

```env
OPENAI_API_KEY=sk-xxx
OPENAI_MODEL=deepseek-chat
OPENAI_BASE_URL=https://api.deepseek.com/v1
OPENAI_TIMEOUT_MS=60000
```

注意：

- RepoPulse 会发送 `response_format: { "type": "json_object" }`，并在 prompt 中要求只输出严格 JSON。
- DeepSeek JSON Output 需要模型上下文里明确出现 JSON 约束；当前 prompt 已满足，但如果你改 prompt，不要删掉“严格 JSON / 不要 Markdown / 不要额外文字”的要求。
- 如果 provider 不支持 `response_format`，通常会显示 `http_error`；这时先查看 debug artifact 中的 `response.raw_response`。
- 调试 OpenAI-compatible provider 时先小批量跑，不要直接全量：

```bash
npm run report -- --profile config/profiles/ai-builder.json --max-candidates 20 --max-analyze 3 --limit 3
```

### AI debug artifact

当模型返回坏 JSON、schema 不符合、HTTP 错误、超时或限流时，可以打开本地调试产物：

```env
REPOPULSE_DEBUG_AI=1
REPOPULSE_DEBUG_DIR=data/debug-ai
```

开启后，AI 失败会写入类似文件：

```text
data/debug-ai/2026-05-18-owner-repo-attempt-1-parse_error.json
```

文件会保存：

- 模型原始响应：`response.raw_response`
- `choices[0].message.content`：`response.message_content`
- 输入摘要：`input_summary`
- 错误类型：`failure_type`
- 错误栈：`error.stack`

不会保存 API key。

验证：

```bash
npm run doctor
npm run report -- --profile config/profiles/ai-builder.json --max-analyze 5 --limit 3
```

如果日报运行概览里显示：

```text
AI/规则分析：openai 5
```

说明已经在使用 OpenAI。

如果显示：

```text
AI/规则分析：heuristic 5
```

说明 OpenAI 未配置或调用失败，系统自动降级到了本地规则。

如果单个项目发生 AI 失败，日报运行概览会额外显示：

```text
AI 成功数：2
heuristic 降级数：1
AI 失败类型分布：parse_error 1
```

对应项目下也会出现：

```text
分析方式：heuristic 降级（AI 失败类型：parse_error；尝试 2 次；debug：data/debug-ai/...json）
```

这说明坏 JSON 没有阻断日报生成，你可以打开 debug 文件查看原始模型输出。

## 3. 优化 AI 分析 prompt

Prompt 不应只要求“总结项目”，而要约束模型做学习决策。

当前 prompt 文件：

```text
src/ai/prompt.js
```

优化重点：

- 明确 RepoPulse 的目标是“学习决策”，不是“热度复述”。
- 要求模型只能基于输入证据判断。
- 要求输出严格 JSON。
- 要求拆分学习价值维度。
- 要求每条理由都带证据。
- 要求低置信度时主动降级语气。

每次改 prompt 后，要运行：

```bash
npm test
npm run report -- --max-analyze 5 --limit 3
```

然后人工检查：

- 是否幻觉项目能力。
- 是否给出具体阅读路径。
- 是否解释了为什么适合当前画像。
- 学习价值明细是否合理。

## 4. 学习价值评分明细

学习价值不再只是一个总分，而是拆成 7 个维度：

| 维度 | 权重 | 含义 |
| --- | --- | --- |
| 文档清晰度 | 20% | README、quickstart、usage、examples 是否充分 |
| 工程结构可读性 | 20% | 是否能看出模块、核心路径、架构线索 |
| 技术代表性 | 15% | 是否代表某类技术趋势或典型工程模式 |
| 实战迁移价值 | 15% | 是否适合做 demo、工具、简历项目或应用封装 |
| 成熟度信号 | 10% | stars、forks、license、维护活跃度 |
| 用户目标匹配 | 10% | 是否匹配语言、topic、学习目标 |
| 综合判断 | 10% | 综合上下文后的兜底判断 |

日报中会展示类似：

```text
学习价值明细：文档清晰度 82 x20%；工程结构可读性 70 x20%；...
```

对应实现：

```text
src/ai/heuristicAnalyzer.js
src/ai/schema.js
src/reports/renderMarkdown.js
```

## 5. 20 个项目人工评测样本

人工评测用于校准评分和 prompt。样本文件：

```text
config/evaluation/manual-review-20.json
```

使用方法：

1. 从样本中选一个用户画像，例如 `ai-builder` 或 `java-backend`。
2. 跑一次日报，或者手动打开样本中的 GitHub 项目。
3. 对每个项目人工填写：
   - 摘要是否准确。
   - 推荐理由是否有证据。
   - 学习价值分是否偏高或偏低。
   - 阅读路径是否可执行。
   - 风险是否合理。
4. 记录 `human_learning_score` 和 `notes`。
5. 对比 RepoPulse 输出，调整 prompt 或评分权重。

### 朋友快评版

如果找朋友评测，不要让对方从“这个 GitHub 项目好不好”的角度评分，而要明确：

> 评分对象是 RepoPulse 的推荐判断是否可信、是否让你更容易决定要不要点开项目。

2026-05-18 的第一轮朋友快评显示，最影响点开的字段是“推荐理由”和“为什么值得关注”；低分和文字反馈主要集中在“风险识别”和“阅读路径”。因此下一轮快评应更轻量，优先问能直接回流到产品改进的问题。

每个项目建议只问 4 个问题：

1. 看完这段，你想点开项目吗？1 到 5 分。
2. 最影响你判断的字段是什么？从“为什么值得关注 / 推荐理由 / 风险 / 学习价值 / 事实来源 / 评分”中选一个。
3. 最大问题是什么？一句话即可。
4. 哪一部分最啰嗦，或者你最希望提前看到什么？一句话即可。

可选补充：

- 哪个风险没有说清楚，例如 API 不稳定、隐私安全、license、外部服务依赖、项目庞大或学习曲线。
- 哪个推荐看起来过强，应该降级为“值得观察”。
- 哪个项目真的会被加入阅读、收藏或复刻清单。

人工评测建议标准：

| 检查项 | 通过标准 |
| --- | --- |
| 摘要准确 | 没有明显事实错误 |
| 推荐理由 | 至少 2 条能从 README / topics / 元数据找到证据 |
| 学习价值分 | 和人工判断差距不超过 15 分 |
| 阅读路径 | 能指导用户下一步打开哪个部分 |
| 风险 | 至少指出文档、维护、license、早期项目、API 稳定性、隐私安全、外部依赖或学习成本等真实风险 |
| 点开意愿 | 用户看完后能明确判断是否值得点开 |

评测汇总除了平均分，还应记录：

- 点开意愿均值。
- 最常见的决策驱动字段。
- 高频风险类型。
- 高频“啰嗦字段”。
- 推荐过强或分层不合理的项目列表。

## 推荐执行顺序

1. 先配置 `GITHUB_TOKEN`，保证数据采集稳定。
2. 再配置 OpenAI Key，提升语义分析质量。
3. 跑小批量报告，人工检查输出。
4. 使用朋友快评版先确认“哪些字段推动用户点开”。
5. 使用 20 个样本做标准基线评测。
6. 根据评测结果再改 prompt、评分权重、风险规则和日报卡片顺序。
