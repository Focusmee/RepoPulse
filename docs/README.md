# RepoPulse 文档使用指南

创建日期：2026-05-15

## 使用原则

1. 先定边界，再写代码。任何功能如果无法对应到产品定位、MVP 或路线图，需要先补文档再实现。
2. 先定契约，再接模型。AI 输出必须先满足结构化契约，再考虑文案优美。
3. 先保闭环，再扩展来源。RepoPulse 的首要目标是稳定地产出有判断价值的日报，不是覆盖所有开源信息源。
4. 评分必须可解释。任何项目进入日报，都要说清楚为什么值得学、适合谁学、风险是什么。
5. 重大变更要留下 ADR。包括评分权重、数据源、模型、数据库结构、日报形态、任务调度方式。

## 推荐工作流

### 新增功能

1. 对照 `01-产品定位与边界.md` 判断是否属于 RepoPulse。
2. 对照 `02-MVP需求说明.md` 判断是否属于当前阶段。
3. 如果改变评分、数据、AI 输出或架构，先更新对应规范文档。
4. 在 `09-迭代路线图与决策记录.md` 追加一条 ADR。
5. 实现后按 `08-验收标准与测试计划.md` 验收。

### 修改评分逻辑

1. 先改 `03-评分模型规范.md`。
2. 明确权重变化、样例影响和预期收益。
3. 补充至少 5 个代表性项目样例，说明新旧排序差异。
4. 通过验收后再改代码。

### 修改 AI 提示词或模型

1. 先改 `05-AI分析与提示词契约.md`。
2. 保证输出 JSON schema 不破坏下游日报和 Web 展示。
3. 抽样复核摘要事实、推荐理由和阅读路径。
4. 记录模型、成本、失败率、质量变化。

### 准备一个版本发布

1. 读 `02-MVP需求说明.md` 确认范围。
2. 按 `08-验收标准与测试计划.md` 跑完整验收。
3. 在 `09-迭代路线图与决策记录.md` 记录版本结果。
4. 如果有偏离文档的地方，先更新文档再发布。

## 文档维护频率

| 文档类型 | 更新频率 |
| --- | --- |
| 产品定位 | 很少改，只有方向变化时改 |
| MVP 需求 | 每个阶段开始和结束时改 |
| 评分模型 | 每次调整权重、维度或个性化规则时改 |
| 数据治理 | 每次新增数据源、字段、采集策略时改 |
| AI 契约 | 每次改 prompt、模型、输出字段时改 |
| 架构边界 | 每次重构、引入新模块或新服务时改 |
| 日报规范 | 每次改内容结构、投递渠道或展示方式时改 |
| 验收计划 | 每次发现新风险、新失败模式时改 |
| 路线图与 ADR | 每周规划和每个重大决策后改 |
| 产品化与变现路线 | 每次改变网站、订阅、付费层级或 SaaS 化判断时改 |
