# AI 分析与提示词契约

## AI 的职责

AI 在 RepoPulse 中负责：

- 总结项目用途。
- 判断项目为什么值得关注。
- 评估学习价值。
- 解释适合哪些人学习。
- 估计当前用户投入该项目的学习成本。
- 给出阅读路径。
- 标出风险。

AI 不负责：

- 伪造事实。
- 替代 GitHub 元数据。
- 在缺少证据时强行给高分。
- 输出不可解析的自由文本作为系统数据。

## 输入契约

每次 AI 分析一个仓库时，输入应包含：

```json
{
  "repo": {
    "full_name": "owner/name",
    "description": "string",
    "language": "string",
    "topics": ["string"],
    "stars": 0,
    "forks": 0,
    "open_issues": 0,
    "created_at": "date",
    "pushed_at": "date",
    "html_url": "string"
  },
  "trend": {
    "stars_1d": 0,
    "stars_7d": 0,
    "forks_7d": 0,
    "source_tags": ["github_trending", "search:topic:agent"]
  },
  "documents": {
    "readme_excerpt": "string",
    "release_excerpt": "string"
  },
  "user_profile": {
    "preferred_languages": ["string"],
    "interested_topics": ["string"],
    "learning_goals": ["string"],
    "excluded_topics": ["string"],
    "skill_level": "junior | intermediate | senior",
    "known_stack": ["string"],
    "weak_areas": ["string"],
    "time_budget": "quick-scan | weekend | deep-study",
    "preferred_project_size": "small | medium | large",
    "goal_priority": ["resume_project", "learn_architecture", "ship_demo", "follow_trend"]
  }
}
```

## 输出契约

AI 必须输出可解析 JSON。禁止在 JSON 外附加说明。

```json
{
  "schema_version": "1.0",
  "summary": "一句话说明项目用途",
  "problem_solved": "它解决的问题",
  "why_it_matters_now": "为什么现在值得关注",
  "learning_value": {
    "score": 0,
    "level": "low | medium | high",
    "breakdown": [
      {
        "id": "documentation_quality",
        "label": "文档清晰度",
        "weight": 0.2,
        "score": 0,
        "reason": "为什么给这个分",
        "evidence": "来自输入的证据"
      }
    ],
    "reasons": [
      {
        "reason": "推荐理由",
        "evidence": "来自 README、元数据或 release 的证据"
      }
    ]
  },
  "trend_explanation": {
    "score_hint": "高 | 中 | 低",
    "signals": ["近期 star 增长", "fork 增长", "release 活跃"]
  },
  "audience": ["适合的人群"],
  "profile_fit": {
    "score": 0,
    "why_for_this_user": "为什么适合当前用户画像"
  },
  "learning_cost": {
    "level": "low | medium | high",
    "score": 0,
    "estimated_effort": "30 分钟了解 / 半天跑通 / 2-3 天深入",
    "prerequisites": ["需要补齐的前置知识"],
    "blockers": ["可能卡住用户的点"],
    "why_for_this_user": "结合当前用户技术情况解释学习成本"
  },
  "recommended_reading_path": [
    {
      "step": 1,
      "action": "先读 README 的某部分",
      "goal": "理解项目目标"
    }
  ],
  "project_idea": "可转化的应用或简历项目想法",
  "risks": [
    {
      "risk": "潜在风险",
      "severity": "low | medium | high"
    }
  ],
  "confidence": {
    "score": 0,
    "reason": "置信度原因"
  }
}
```

## 字段要求

| 字段 | 要求 |
| --- | --- |
| `summary` | 不能超过 40 个中文字 |
| `learning_value.score` | 0 到 100 的整数 |
| `learning_value.breakdown` | 必须包含 7 个学习价值分项，每项包含分数、权重、原因和证据 |
| `learning_value.reasons` | 2 到 4 条，每条必须有 evidence |
| `audience` | 1 到 4 个具体人群 |
| `learning_cost.level` | low / medium / high，必须结合用户技术情况和项目复杂度 |
| `learning_cost.score` | 0 到 100 的整数，越高表示越适合当前用户投入 |
| `learning_cost.estimated_effort` | 必须是可理解的时间投入描述 |
| `learning_cost.prerequisites` | 列出关键前置知识，没有则为空数组 |
| `learning_cost.blockers` | 列出可能卡住用户的环境、依赖、源码规模或概念门槛 |
| `recommended_reading_path` | 2 到 5 步，必须可执行 |
| `risks` | 至少 1 条；没有明显风险也要说明“不确定性” |
| `confidence.score` | 0 到 100 的整数 |

## 证据规则

AI 给出的判断必须来自输入中的证据：

- README。
- 仓库描述。
- topics。
- language。
- stars / forks / issue / release。
- 用户画像。
- 用户已掌握技术栈、薄弱项、时间预算和项目规模偏好。

如果证据不足，必须降低 `confidence.score`，并在风险中说明。

学习成本必须和风险区分开：

- 风险描述项目自身的不确定性，例如 license 不明、维护弱、API 不稳定。
- 学习成本描述当前用户投入该项目的门槛，例如环境复杂、源码规模大、依赖陌生技术栈。

## 禁止行为

AI 禁止：

- 根据项目名猜测不存在的功能。
- 声称项目有文档但 README 未体现。
- 把 awesome list 当作完整工程项目。
- 把 demo / toy project 说成生产级框架。
- 输出无法解析的 Markdown。
- 输出“非常值得学习”但不给证据。

## Prompt 结构

建议 prompt 固定分为：

1. 系统角色：你是开源项目学习价值分析器。
2. 任务目标：评估项目是否值得当前用户学习。
3. 输入数据：repo、trend、documents、user_profile。
4. 输出 schema：严格 JSON。
5. 证据约束：只能基于输入判断。
6. 质量要求：具体、可执行、可解释。

## 输出校验

AI 输出后必须进行程序校验：

- JSON 是否可解析。
- `schema_version` 是否支持。
- 必填字段是否存在。
- 分数是否在 0 到 100。
- reasons 是否包含 evidence。
- reading_path 是否为空。
- confidence 过低时是否进入降级流程。

校验失败时：

1. 如果是 `parse_error`，使用更短的 README / release 输入和更强 JSON 约束重试一次。
2. 如果重试仍失败，降级为本地 heuristic 分析。
3. 记录 AI 失败类型：`parse_error`、`schema_error`、`http_error`、`timeout`、`rate_limit`。
4. 不应阻塞整份日报生成。
5. 日报必须展示 AI 成功数、heuristic 降级数、AI 失败类型分布。

开启调试：

```env
REPOPULSE_DEBUG_AI=1
REPOPULSE_DEBUG_DIR=data/debug-ai
```

调试产物必须包含：

- 模型原始响应。
- `choices[0].message.content`。
- 输入摘要。
- 错误类型。
- 错误栈。

调试产物不得包含 API key。

## 低置信降级

当 `confidence.score < 60` 时：

- 日报中不应给强推荐语气。
- 可放入“值得观察”而不是“今日最值得深读”。
- 必须展示风险或不确定性。
- 后续可尝试补充 release、docs、examples 后重新分析。

## 质量抽检

每次修改 prompt 或模型后，至少抽查：

- 5 个高热项目。
- 5 个中等热度但文档优秀项目。
- 5 个 README 很弱或项目早期的项目。
- 5 个用户强相关项目。

观察：

- 是否幻觉。
- 是否理由具体。
- 是否阅读路径可执行。
- 是否对用户画像有真实响应。
