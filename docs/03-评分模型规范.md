# 评分模型规范

## 评分目标

RepoPulse 的评分不是为了找“最火项目”，而是为了找“当前最值得某个用户学习的项目”。

评分系统分三层：

1. `trend_score`：项目近期上升势头。
2. `learning_score`：项目作为学习材料的价值。
3. `profile_match_score`：项目与用户语言、topic、目标的匹配程度。
4. `learning_cost_score`：项目对当前用户的投入成本是否可接受。
5. `personalized_score`：结合用户画像后的最终推荐分。

所有分数统一使用 0 到 100 分。

## 评分原则

1. 热度是信号，不是结论。
2. 学习价值必须有证据支撑。
3. 个性化排序必须能解释“为什么适合你”。
4. 权重调整必须记录原因和样例影响。
5. 不允许只有总分，没有分项和理由。
6. 学习成本必须结合用户技术情况判断，不能把“项目很复杂”简单等同于“项目不好”。

## 候选项目基础过滤

进入评分前先过滤明显不适合项目：

| 规则 | 建议 |
| --- | --- |
| 仓库不可访问 | 过滤 |
| README 缺失且描述极短 | 降权或过滤 |
| 最近 12 个月无更新 | 降权，除非属于经典项目 |
| 明显模板、awesome 列表、资源合集 | 默认降权，除非用户画像明确需要 |
| 与用户排除主题匹配 | 过滤或强降权 |
| 重复仓库 | 保留来源更多或数据更新的一条 |

## 热度分 `trend_score`

热度分衡量项目是否正在被更多人关注。初始公式：

```text
trend_score =
  0.40 * normalized_7d_star_growth
+ 0.20 * normalized_1d_star_growth
+ 0.15 * normalized_fork_growth
+ 0.15 * normalized_activity_signal
+ 0.10 * recency_score
```

### 分项定义

| 分项 | 含义 |
| --- | --- |
| `normalized_7d_star_growth` | 近 7 日 star 增量的标准化分 |
| `normalized_1d_star_growth` | 近 1 日 star 增量的标准化分 |
| `normalized_fork_growth` | fork 增长，代表动手意愿 |
| `normalized_activity_signal` | issue、PR、release、commit 等活跃度综合信号 |
| `recency_score` | 最近更新时间，越新越高，但不鼓励无意义刷 commit |

### 标准化要求

不同语言和生态的 star 基数差异很大，不能直接比较绝对值。首版可使用每日候选集内 percentile 或 min-max 标准化；后续可按语言 / topic 分桶标准化。

## 学习价值分 `learning_score`

学习价值分是 RepoPulse 的核心差异化。初始公式：

```text
learning_score =
  0.20 * documentation_quality
+ 0.20 * code_structure_readability
+ 0.15 * technical_representativeness
+ 0.15 * practical_transfer_value
+ 0.10 * maturity_signal
+ 0.10 * profile_goal_fit
+ 0.10 * ai_overall_judgement
```

### 分项定义

| 分项 | 判断问题 | 证据 |
| --- | --- | --- |
| `documentation_quality` | README 是否清晰说明用途、安装、示例和架构 | README、docs、examples |
| `code_structure_readability` | 目录结构是否适合阅读和学习 | 文件结构、模块命名、examples |
| `technical_representativeness` | 是否代表某类技术趋势或经典工程模式 | topics、依赖、README 描述 |
| `practical_transfer_value` | 是否能迁移到求职项目、业务应用或个人工具 | use cases、examples、license |
| `maturity_signal` | 是否有 issue、release、维护节奏和社区反馈 | release、issues、commits |
| `profile_goal_fit` | 是否匹配用户语言、主题和学习目标 | 用户画像配置 |
| `ai_overall_judgement` | 综合上下文后的主观判断 | AI 结构化分析，但必须引用证据 |

### 学习价值解释

每个项目必须输出：

- 总分。
- 至少 3 个分项分。
- 2 到 4 条学习价值理由。
- 至少 1 条风险或不推荐理由。
- 推荐阅读路径。

## 个性化分 `personalized_score`

最终排序建议使用：

```text
personalized_score =
  0.25 * trend_score
+ 0.40 * learning_score
+ 0.20 * profile_match_score
+ 0.10 * learning_cost_score
+ 0.10 * novelty_score
- penalty_score
```

### 分项说明

| 分项 | 含义 |
| --- | --- |
| `profile_match_score` | 语言、topic、学习目标、排除项的匹配程度 |
| `learning_cost_score` | 当前用户投入该项目是否划算；分数越高表示成本越可接受 |
| `novelty_score` | 对用户来说是否新鲜，避免每天重复推荐同一批项目 |
| `penalty_score` | README 缺失、项目停更、license 不明、AI 低置信等惩罚 |

## 学习成本分 `learning_cost_score`

学习成本分回答的问题是：**以当前用户的技术情况和时间预算，这个项目是否值得现在投入？**

它不替代 `learning_score`。一个项目可以学习价值很高，但对当前用户成本也很高；这种项目更适合进入“值得观察”或“项目灵感”，而不是直接强推荐。

### 用户画像扩展

学习成本需要额外读取以下画像字段：

| 字段 | 含义 |
| --- | --- |
| `skill_level` | junior / intermediate / senior |
| `known_stack` | 已掌握技术栈，例如 TypeScript、Python、Docker、React |
| `weak_areas` | 薄弱或暂不想投入的方向，例如 Kubernetes、Rust |
| `time_budget` | quick-scan / weekend / deep-study |
| `preferred_project_size` | small / medium / large |
| `goal_priority` | resume_project / learn_architecture / ship_demo / follow_trend |

### 初始判断规则

| 信号 | 影响 |
| --- | --- |
| 项目语言命中 `known_stack` | 降低学习成本 |
| topics / README 命中 `weak_areas` | 提高学习成本，除非目标是补齐该能力 |
| README 有 quickstart、examples、demo | 降低学习成本 |
| 依赖 Docker、Kubernetes、GPU、本地模型、大型 monorepo | 提高学习成本 |
| 项目规模大、模块多、issue 多 | 提高学习成本 |
| 有清晰 docs、examples、template | 降低学习成本 |
| `time_budget=quick-scan` 且项目需要复杂部署 | 降级为观察 |
| `goal_priority=resume_project` 且可拆成小 demo | 即使成本中等也可进入项目灵感 |

### 分层影响

- 高学习价值 + 低/中学习成本：优先进入“今日最值得深读”。
- 高热度 + 高学习成本：进入“上升很快，值得观察”。
- 高学习成本 + 高复刻价值：进入“可转化为项目灵感”，并说明拆解路径。
- 高学习成本 + 低画像匹配：进入“谨慎关注”或直接不推荐。
- 高学习成本项目不能标为“强推荐”，除非同时说明投入收益、前置知识和最小行动路径。

## 推荐分层

日报中不建议只给一个总榜。可以按推荐意图分层：

| 层级 | 含义 |
| --- | --- |
| 今日最值得深读 | 学习价值高，和用户目标强匹配 |
| 上升很快，值得观察 | trend_score 高，但学习价值或成熟度未完全确认 |
| 可转化为项目灵感 | practical_transfer_value 高 |
| 谨慎关注 | 热度高但风险明显 |

### 分层后处理规则

排序完成后必须进行确定性后处理，避免 AI 文案或单一分数导致过度推荐：

1. “强推荐”最多 3 个，且必须位于全局 Top 3、`personalized_score >= 75`、`learning_score >= 80`、`confidence.score >= 75`，没有 high risk。
2. “今日最值得深读”最多 3 个，且必须同时满足学习价值高、画像匹配强、证据充分、风险可接受。
3. 明显 `awesome`、资源合集、示例集合、starter、template、boilerplate 类项目默认不进入“今日最值得深读”，应进入“可转化为项目灵感”或其他较保守分区。
4. “可转化为项目灵感”不能只因为 AI 生成了 `project_idea` 就进入；除资源/模板/示例类项目外，还必须有高 `practical_transfer_value`、足够证据和画像相关性。
5. “上升很快，值得观察”用于承接热度或增长明显，但学习价值、成熟度、证据或画像匹配仍未完全确认的项目。
6. high risk、低置信度或重大惩罚项项目默认进入“谨慎关注”。

### 仓库类型识别

仓库类型识别应优先使用事实字段：

- `full_name`
- `description`
- `topics`
- AI 输出中的 `summary` / `problem_solved`

不得使用 `project_idea` 作为仓库类型判断依据，因为它是模型生成的转化建议，不是仓库事实。

## 权重调整流程

任何权重调整必须完成：

1. 说明调整动机。
2. 给出调整前后公式。
3. 选取至少 5 个样例项目比较排序变化。
4. 说明是否影响日报解释文案。
5. 在 `09-迭代路线图与决策记录.md` 追加 ADR。

## 评分反例

这些情况应被识别：

- 只有营销型 README，没有可运行示例。
- star 快速增长但项目几乎没有源码。
- awesome 列表被误判为工程项目。
- 项目很火但和用户目标完全不相关。
- AI 根据项目名臆测功能。

## 初始验收标准

1. 学习价值分和热度分不能高度同质化。
2. 高热度低学习价值项目应能被降权。
3. 中等热度但文档清晰、工程结构优秀的项目应能进入推荐。
4. 每个分数必须有理由和证据字段。
