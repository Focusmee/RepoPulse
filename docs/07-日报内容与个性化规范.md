# 日报内容与个性化规范

## 日报目标

日报不是信息堆砌，而是帮助用户做学习决策。读完日报后，用户应该能决定：

- 今天优先看哪 1 到 3 个项目。
- 这个项目现在值不值得点开。
- 为什么现在值得关注。
- 为什么它们适合自己。
- 对当前用户来说学习成本是否可接受。
- 最大风险是什么。
- 点开后第一步做什么。
- 是否值得做成 demo、简历项目或产品想法。

2026-05-18 朋友快评反馈显示，最影响用户点开的字段是“推荐理由”和“为什么值得关注”，最薄弱的字段是“风险识别”和“阅读路径”。后续日报应优先帮助用户在 30 秒内判断“要不要点开”，而不是平均展示所有字段。

## 精华版日报结构

用于社交平台、私信试读、订阅页和早期成交时，应从完整 Markdown 日报生成一份短版 HTML 精华版。精华版不是替代完整日报，而是推广和转化层。

精华版必须满足：

- 5 分钟内能读完。
- Top 3 项目在 30 秒内能判断是否值得点开。
- 只保留高转化字段：一句话定位、是否值得点开、为什么值得关注、推荐理由、学习成本、最大风险、推荐动作和项目想法。
- 最多展示 3 个深读项目和 2 个项目灵感。
- 保留 GitHub 链接和查看完整版入口。
- HTML 内置移动端和打印样式；PDF 首版通过浏览器打印导出，不单独维护自动 PDF 生成链路。

推荐 HTML 精华版结构：

```text
封面：日期、画像、今日结论、候选数、分析数、推荐数、质量警告
Top 3 值得点开的项目
  - 项目名与链接
  - 一句话定位
  - 项目背景与使用场景
  - 简单例子
  - 是否值得点开
  - 为什么值得关注
  - 推荐理由前 2 条
  - 学习成本摘要
  - 最大风险前 2 条
  - 推荐动作前 2 条
  - 可转化项目想法
可转化为项目灵感：最多 2 个
CTA：申请试读 / 订阅 AI Builder Radar / 查看完整版
```

## 单项目卡片字段

每个项目必须包含：

| 字段 | 要求 |
| --- | --- |
| 项目名 | 使用 `owner/name` |
| 一句话定位 | 不超过 40 个中文字 |
| 项目背景与使用场景 | 用 120 到 180 字解释它是什么、解决什么问题、谁会在什么场景使用；必须让不熟悉该技术的人也能看懂 |
| 简单例子 | 用一个专业或形象的小例子说明项目发挥作用，避免抽象口号 |
| 能学习到什么 | 说明可学习的工程能力、技术模式或可复刻点，例如 Agent 工作流、RAG 架构、浏览器自动化、CLI 设计 |
| 推荐等级 | 强推荐 / 值得观察 / 项目灵感 / 谨慎关注 |
| 项目类型 | 说明是应用/库、框架/平台、开发者工具、资源合集、模板或示例集合 |
| 综合分 | 展示综合分、学习价值、热度和画像匹配 |
| 置信度 | 展示 AI/规则分析置信度和原因 |
| 为什么值得关注 | 展示关键增长指标、社区信号和近期活跃，不堆全部数据 |
| 推荐理由与证据 | 每条理由必须拆成“判断”和“证据” |
| 学习成本 | 展示为低成本 / 中等成本 / 高成本；内部仍使用 low / medium / high |
| 最大风险 | 最多 3 条，必须具体，优先覆盖 API 稳定性、隐私安全、license、外部依赖、维护节奏、项目规模和环境复杂度 |
| 推荐动作 | 1 到 3 个下一步动作，例如先看 README 哪一节、clone 后让 AI 分析哪些入口/模块 |
| 为什么适合当前画像 | 必须关联语言、topic 或学习目标 |
| 学习价值 | 展示分数、分项和 2 到 4 条理由；默认后置，不抢占首屏判断 |
| 项目想法 | 可选，但对求职和应用开发很有价值 |
| 事实来源 | 展示 GitHub metadata、README、release、topics、license；作为证据脚注后置 |
| 链接 | GitHub URL |

## 单项目卡片优先级

面向阅读体验时，单项目卡片应按“决策优先”排序：

1. **一句话定位**：先说明项目是什么。
2. **项目背景与使用场景**：解释它解决什么问题、谁会用、在什么场景用。
3. **简单例子**：用一个具体例子让不熟悉该技术的人也能理解它的作用。
4. **是否值得点开**：一句话给出结论，避免只说“值得学习”。
5. **为什么值得关注**：解释当前趋势信号、社区信号和近期活跃。
6. **推荐理由与证据**：说明它为什么适合当前画像，判断必须带证据。
7. **能学习到什么 / 可复刻点**：突出学习收益、demo 方向或产品机会。
8. **学习成本**：说明对当前用户来说是低/中/高成本，需要哪些前置知识，预计投入多久。
9. **最大风险**：最多 3 条，必须具体，不用泛泛说“项目较新”。
10. **推荐动作**：点开后先看什么；如果适合 clone，应说明可以让 AI 分析哪些文件、模块或风险。
11. **事实来源和学习价值明细**：保留可复核性，但默认后置。

不推荐把事实来源、长评分明细和 5 步阅读路径放在推荐理由之前。它们能增强可信度，但不是朋友快评中最影响点开的信息。

## 语气规范

日报语气应：

- 直接。
- 有判断。
- 不夸张。
- 不营销。
- 不用空泛形容词堆砌。

避免：

- “这是一个非常优秀的项目”但不给理由。
- “值得学习”但不说明适合谁。
- “可以提升技术能力”这种泛泛表达。
- 长篇复制 README。
- 长篇铺开事实来源，却没有先说为什么值得点开。
- 风险只写“项目较新”“学习成本高”，但不说明具体来源或影响。
- 把“对当前用户偏难”直接写成项目风险；应改写为学习成本，并说明触发信号，例如项目规模大、环境复杂、依赖陌生技术栈。
- 不直接使用 low / medium / high 等英文枚举作为用户可见文案；展示层应翻译成低 / 中 / 高、低风险 / 中风险 / 高风险。
- 声称已经读过源码或确认源码结构，除非系统确实读取了目录树或关键文件；基于 README 的判断必须写成“README 显示”“文档暗示”“建议 clone 后分析”。

## 展示层中文化

内部 schema 可以继续使用稳定枚举值，但用户可见报告必须中文化：

| 内部值 | 学习成本展示 | 风险展示 | 学习价值展示 |
| --- | --- | --- | --- |
| `low` | 低成本 | 低风险 | 低 |
| `medium` | 中等成本 | 中风险 | 中 |
| `high` | 高成本 | 高风险 | 高 |

质量警告等级也应展示为中文，但可以保留英文类型代码便于调试，例如：`低 / overselling_language`。

## 项目背景、使用场景和例子

为了让技术不熟悉的读者一眼理解项目价值，每个 Top 项目都应补充：

- **项目背景与使用场景**：120 到 180 字，回答“它是什么、为什么需要它、谁会用它、什么时候用它”。
- **简单例子**：用一个具体场景说明项目发挥作用，例如“如果你要做一个 AI 自动收集竞品价格的 Agent，这个项目负责解决网页自动化被拦截的问题”。
- **能学习到什么**：说明读这个项目能学到的能力，例如浏览器自动化、Agent 工程化、RAG 记忆、CLI 设计、工作流编排、部署安全。

这三个字段的目标是提高理解速度和付费转化，不是复述 README。它们必须短、具体、有场景感。

## 源码分析边界

当前基础日报默认基于 README、release 和 GitHub 元数据判断，不等同于源码级审计。报告中必须避免暗示系统已经阅读完整源码。

后续可规划“Top 项目源码轻扫”作为深度版能力：

- 只对 Top 1 到 3 项目启用。
- 先读取目录树和关键入口文件，不 clone 全仓库。
- 优先读取 `package.json`、`pyproject.toml`、`README`、`examples`、`src/index.*`、`main.py`、`packages/*` 等低成本入口。
- 输出项目入口、模块边界、可复刻点、源码阅读路径和源码级风险。
- 不作为基础日报前置条件，而是作为付费深度版或后续增强卖点。

### 多平台推广画像

面向变现和内容分发时，首批只维护 4 个核心画像，避免按行业过早拆散产品定位：

| 画像 | 定位 | 主要用途 |
| --- | --- | --- |
| `ai-builder` | AI 应用开发者 / 独立开发者 | 核心付费画像，主推 AI Builder Radar |
| `ai-career-builder` | 想做 AI 项目的求职开发者 | 小红书、抖音、B 站内容中展示简历项目和学习路径 |
| `ai-product-operator` | 懂一点技术的产品 / 运营 / 增长人员 | 展示 AI 工具、自动化工作流和业务效率机会 |
| `industry-ai-explorer` | 想用 AI 改造业务的行业从业者 | 做泛行业获客，强调可外包、可合作、可落地的方向 |

产品成交优先服务 `ai-builder`。其余画像主要用于内容选题、社交平台破圈和需求探索，只有当试读反馈证明有付费意愿时，才继续细分行业画像。

## 个性化规则

日报必须体现画像：

- 语言匹配：偏好语言项目获得加分。
- 主题匹配：关注 topic 获得加分。
- 目标匹配：求职、架构学习、应用开发分别影响推荐理由。
- 技术水平匹配：同一项目对不同 `skill_level` 的学习成本不同，不能只按项目热度判断。
- 已掌握技术栈匹配：项目依赖用户已掌握的 `known_stack` 时学习成本降低。
- 薄弱项约束：项目核心依赖 `weak_areas` 时应降低推荐强度，除非目标是刻意补齐该能力。
- 时间预算匹配：`quick-scan` 用户优先推荐易判断、易跑通项目；`deep-study` 用户可以接受复杂源码。
- 项目规模偏好：大型平台项目不应默认推荐给偏好 small demo 的用户。
- 排除项：命中排除 topic 的项目降权或过滤。
- 新鲜度：近期已推荐过的项目降低重复出现概率。

## 学习成本解释

学习成本不是项目风险，而是“当前用户投入这个项目的成本”。日报应显式输出：

| 字段 | 要求 |
| --- | --- |
| `level` | low / medium / high |
| `investment_fit_score` | 0 到 100，越高表示越适合当前用户现在投入，不表示成本越高 |
| `estimated_effort` | 例如 30 分钟了解、半天跑通、2-3 天深入 |
| `prerequisites` | 需要补齐的前置知识，例如 Docker、RAG、MCP、Rust |
| `blockers` | 可能卡住用户的点，例如依赖外部 API、大型 monorepo、环境复杂 |
| `why_for_this_user` | 结合当前画像解释为什么成本低/中/高 |

实现约束：

- `learning_cost` 必须由统一规则模块提供兜底，不能只依赖 AI 输出。
- AI 输出可以补充解释和前置知识，但缺失或校验失败时必须回落到本地规则估算。
- `level` 表示成本高低；`investment_fit_score` 表示当下投入适配度，两者不能混用。
- 风险和学习成本要分开：风险写项目自身不确定性，学习成本写当前用户投入门槛。

推荐语气示例：

- 低成本：`对当前画像成本较低，已有 TypeScript 和 agent 背景，建议先跑 quickstart。`
- 中成本：`需要补 Docker Compose 和 RAG 基础，适合周末做一个最小 demo。`
- 高成本：`项目规模较大且依赖多项陌生技术，建议先作为趋势观察，不建议直接读全量源码。`

## 不同学习目标的解释重点

| 学习目标 | 推荐理由应强调 |
| --- | --- |
| 找工作 | 技术栈匹配、可写进简历、工程完整度 |
| 学架构 | 模块边界、设计模式、可读性、规模 |
| 读源码 | 目录清晰度、核心路径、测试和 examples |
| 做应用 | 问题场景、可改造空间、部署难度 |
| 跟趋势 | 增长信号、生态位置、相似项目 |

## 分层推荐规则

### 今日最值得深读

进入条件：

- `learning_score` 高。
- `profile_match_score` 高。
- `learning_cost.level` 不是 high，或 high 但收益和行动建议非常明确。
- 证据充分。
- 风险可接受。
- 数量最多 3 个。
- 推荐等级为“强推荐”的项目必须位于全局 Top 3，且不是资源合集、模板或示例集合。

### 上升很快，值得观察

进入条件：

- `trend_score` 高。
- 但项目成熟度、文档或用户匹配度未完全确认。
- 热度高但证据或成熟度仍需继续观察。
- 对当前用户学习成本高，但趋势信号强，适合作为观察对象。

### 可转化为项目灵感

进入条件：

- 实战迁移价值高。
- 能改造成 demo、工具、插件、模板或 SaaS 小功能。
- 即使学习成本较高，也能拆成小 demo 或局部复刻。
- 明显资源合集、awesome list、示例集合、starter、template、boilerplate 项目默认进入该层。
- 不是资源/模板/示例类的项目，不能只因为有 `project_idea` 就进入该层；还需要高实战迁移分、画像相关性和明确证据。

### 谨慎关注

进入条件：

- 热度高但有明显风险。
- 文档不足。
- 项目过早期。
- license 或维护状态不明。

## 质量警告

日报生成后应运行质量检查。质量警告不一定代表日报失败，但必须在报告中显式展示，方便人工复核。

首版质量警告包括：

- 强推荐数量超过 3。
- “今日最值得深读”数量超过 3。
- Top 项目全部落在同一个分区。
- 推荐理由证据为空或过于空泛。
- 出现异常 token，例如 `iii`、`READM`、`undefined`、`null`、`NaN` 或乱码字符。
- 出现过度营销化表达，例如“完美匹配”“极完备”“一应俱全”“最强”“必看”。

报告中应展示：

```markdown
## 质量警告

- medium / suspicious_token：owner/name：分析文本中出现异常 token：READM
```

## 日报长度

首版建议：

- Top 深读项目：最多 3 个。
- 值得观察项目：3 到 5 个。
- 项目灵感：2 到 3 个。
- 总篇幅控制在 10 分钟内能读完。
- 单项目首屏内容控制在 30 秒内能读完，先给推荐理由、值得关注和最大风险。
- 推荐动作控制在 1 到 3 条；完整阅读路径可放到详情页或结构化数据。

## 日报质量底线

一份日报如果出现以下情况，应视为不合格：

- 只列项目，没有判断。
- 推荐理由和用户画像无关。
- 学习价值分没有解释。
- 阅读路径不可执行。
- 风险全部为空。
- 明显复制 README 而没有分析。
- AI 编造项目能力。
- 所有项目都被标为“强推荐”。
- 资源合集、模板或示例集合被误判为生产级深读项目。
- 推荐理由只有判断，没有可复核证据。
