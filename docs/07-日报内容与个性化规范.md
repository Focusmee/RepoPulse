# 日报内容与个性化规范

## 日报目标

日报不是信息堆砌，而是帮助用户做学习决策。读完日报后，用户应该能决定：

- 今天优先看哪 1 到 3 个项目。
- 这个项目现在值不值得点开。
- 为什么现在值得关注。
- 为什么它们适合自己。
- 对当前用户来说学习成本是否可接受。
- 最大风险是什么。
- 点开后第一步做什么。
- 是否值得做成 demo、简历项目或产品想法。

2026-05-18 朋友快评反馈显示，最影响用户点开的字段是“推荐理由”和“为什么值得关注”，最薄弱的字段是“风险识别”和“阅读路径”。后续日报应优先帮助用户在 30 秒内判断“要不要点开”，而不是平均展示所有字段。

推荐 Markdown 结构：

```markdown

## 今日结论

用 3 到 5 句话总结今天的技术趋势和最值得关注方向。

## 今日最值得深读

### 1. owner/name

- 一句话定位：
- 推荐等级：
- 项目类型：
- 综合分：
- 置信度：
- 为什么值得关注：
- 推荐理由与证据：
- 学习成本：
- 最大风险：
- 推荐动作：
- 为什么适合当前画像：
- 为什么值得学习：
- 可转化项目想法：
- 事实来源：
- 学习价值明细：
- 链接：

## 上升很快，值得观察

## 可转化为项目灵感

## 今日不推荐但值得知道
```

## 单项目卡片字段

每个项目必须包含：

| 字段 | 要求 |
| --- | --- |
| 项目名 | 使用 `owner/name` |
| 一句话定位 | 不超过 40 个中文字 |
| 推荐等级 | 强推荐 / 值得观察 / 项目灵感 / 谨慎关注 |
| 项目类型 | 说明是应用/库、框架/平台、开发者工具、资源合集、模板或示例集合 |
| 综合分 | 展示综合分、学习价值、热度和画像匹配 |
| 置信度 | 展示 AI/规则分析置信度和原因 |
| 为什么值得关注 | 展示关键增长指标、社区信号和近期活跃，不堆全部数据 |
| 推荐理由与证据 | 每条理由必须拆成“判断”和“证据” |
| 学习成本 | 估计 low / medium / high、投入时间、前置知识和为什么适合或不适合当前用户 |
| 最大风险 | 最多 3 条，必须具体，优先覆盖 API 稳定性、隐私安全、license、外部依赖、维护和学习成本 |
| 推荐动作 | 1 到 3 个下一步动作，例如先看 README 哪一节、clone 后让 AI 分析哪些入口/模块 |
| 为什么适合当前画像 | 必须关联语言、topic 或学习目标 |
| 学习价值 | 展示分数、分项和 2 到 4 条理由；默认后置，不抢占首屏判断 |
| 项目想法 | 可选，但对求职和应用开发很有价值 |
| 事实来源 | 展示 GitHub metadata、README、release、topics、license；作为证据脚注后置 |
| 链接 | GitHub URL |

## 单项目卡片优先级

面向阅读体验时，单项目卡片应按“决策优先”排序：

1. **是否值得点开**：一句话给出结论，避免只说“值得学习”。
2. **为什么值得关注**：解释当前趋势信号、社区信号和近期活跃。
3. **推荐理由与证据**：说明它为什么适合当前画像，判断必须带证据。
4. **学习成本**：说明对当前用户来说是低/中/高成本，需要哪些前置知识，预计投入多久。
5. **最大风险**：最多 3 条，必须具体，不用泛泛说“项目较新”。
6. **推荐动作**：点开后先看什么；如果适合 clone，应说明可以让 AI 分析哪些文件、模块或风险。
7. **事实来源和学习价值明细**：保留可复核性，但默认后置。

不推荐把事实来源、长评分明细和 5 步阅读路径放在推荐理由之前。它们能增强可信度，但不是朋友快评中最影响点开的信息。

## 语气规范

日报语气应：

- 直接。
- 有判断。
- 不夸张。
- 不营销。
- 不用空泛形容词堆砌。

避免：

- “这是一个非常优秀的项目”但不给理由。
- “值得学习”但不说明适合谁。
- “可以提升技术能力”这种泛泛表达。
- 长篇复制 README。
- 长篇铺开事实来源，却没有先说为什么值得点开。
- 风险只写“项目较新”“学习成本高”，但不说明具体来源或影响。

## 个性化画像

用户画像至少包含：

```yaml
profile_id: default
role: "后端开发者"
preferred_languages:
  - Java
  - TypeScript
interested_topics:
  - distributed-systems
  - llm
  - agent
learning_goals:
  - 找工作
  - 学架构
  - 做可展示项目
skill_level: intermediate
known_stack:
  - TypeScript
  - Python
  - Docker
weak_areas:
  - Kubernetes
time_budget: weekend
preferred_project_size: medium
goal_priority:
  - resume_project
  - ship_demo
excluded_topics:
  - blockchain
daily_limit: 10
```

## 个性化规则

日报必须体现画像：

- 语言匹配：偏好语言项目获得加分。
- 主题匹配：关注 topic 获得加分。
- 目标匹配：求职、架构学习、应用开发分别影响推荐理由。
- 技术水平匹配：同一项目对不同 `skill_level` 的学习成本不同，不能只按项目热度判断。
- 已掌握技术栈匹配：项目依赖用户已掌握的 `known_stack` 时学习成本降低。
- 薄弱项约束：项目核心依赖 `weak_areas` 时应降低推荐强度，除非目标是刻意补齐该能力。
- 时间预算匹配：`quick-scan` 用户优先推荐易判断、易跑通项目；`deep-study` 用户可以接受复杂源码。
- 项目规模偏好：大型平台项目不应默认推荐给偏好 small demo 的用户。
- 排除项：命中排除 topic 的项目降权或过滤。
- 新鲜度：近期已推荐过的项目降低重复出现概率。

## 学习成本解释

学习成本不是项目风险，而是“当前用户投入这个项目的成本”。日报应显式输出：

| 字段 | 要求 |
| --- | --- |
| `level` | low / medium / high |
| `estimated_effort` | 例如 30 分钟了解、半天跑通、2-3 天深入 |
| `prerequisites` | 需要补齐的前置知识，例如 Docker、RAG、MCP、Rust |
| `blockers` | 可能卡住用户的点，例如依赖外部 API、大型 monorepo、环境复杂 |
| `why_for_this_user` | 结合当前画像解释为什么成本低/中/高 |

推荐语气示例：

- 低成本：`对当前画像成本较低，已有 TypeScript 和 agent 背景，建议先跑 quickstart。`
- 中成本：`需要补 Docker Compose 和 RAG 基础，适合周末做一个最小 demo。`
- 高成本：`项目规模较大且依赖多项陌生技术，建议先作为趋势观察，不建议直接读全量源码。`

## 不同学习目标的解释重点

| 学习目标 | 推荐理由应强调 |
| --- | --- |
| 找工作 | 技术栈匹配、可写进简历、工程完整度 |
| 学架构 | 模块边界、设计模式、可读性、规模 |
| 读源码 | 目录清晰度、核心路径、测试和 examples |
| 做应用 | 问题场景、可改造空间、部署难度 |
| 跟趋势 | 增长信号、生态位置、相似项目 |

## 分层推荐规则

### 今日最值得深读

进入条件：

- `learning_score` 高。
- `profile_match_score` 高。
- `learning_cost.level` 不是 high，或 high 但收益和行动建议非常明确。
- 证据充分。
- 风险可接受。
- 数量最多 3 个。
- 推荐等级为“强推荐”的项目必须位于全局 Top 3，且不是资源合集、模板或示例集合。

### 上升很快，值得观察

进入条件：

- `trend_score` 高。
- 但项目成熟度、文档或用户匹配度未完全确认。
- 热度高但证据或成熟度仍需继续观察。
- 对当前用户学习成本高，但趋势信号强，适合作为观察对象。

### 可转化为项目灵感

进入条件：

- 实战迁移价值高。
- 能改造成 demo、工具、插件、模板或 SaaS 小功能。
- 即使学习成本较高，也能拆成小 demo 或局部复刻。
- 明显资源合集、awesome list、示例集合、starter、template、boilerplate 项目默认进入该层。
- 不是资源/模板/示例类的项目，不能只因为有 `project_idea` 就进入该层；还需要高实战迁移分、画像相关性和明确证据。

### 谨慎关注

进入条件：

- 热度高但有明显风险。
- 文档不足。
- 项目过早期。
- license 或维护状态不明。

## 质量警告

日报生成后应运行质量检查。质量警告不一定代表日报失败，但必须在报告中显式展示，方便人工复核。

首版质量警告包括：

- 强推荐数量超过 3。
- “今日最值得深读”数量超过 3。
- Top 项目全部落在同一个分区。
- 推荐理由证据为空或过于空泛。
- 出现异常 token，例如 `iii`、`READM`、`undefined`、`null`、`NaN` 或乱码字符。
- 出现过度营销化表达，例如“完美匹配”“极完备”“一应俱全”“最强”“必看”。

报告中应展示：

```markdown
## 质量警告

- medium / suspicious_token：owner/name：分析文本中出现异常 token：READM
```

## 日报长度

首版建议：

- Top 深读项目：最多 3 个。
- 值得观察项目：3 到 5 个。
- 项目灵感：2 到 3 个。
- 总篇幅控制在 10 分钟内能读完。
- 单项目首屏内容控制在 30 秒内能读完，先给推荐理由、值得关注和最大风险。
- 推荐动作控制在 1 到 3 条；完整阅读路径可放到详情页或结构化数据。

## 归档命名

建议：

```text

    2026-05-15-default.md
```

如果有多个画像：

```text
2026-05-15-java-backend.md

```

## 日报质量底线

一份日报如果出现以下情况，应视为不合格：

- 只列项目，没有判断。
- 推荐理由和用户画像无关。
- 学习价值分没有解释。
- 阅读路径不可执行。
- 风险全部为空。
- 明显复制 README 而没有分析。
- AI 编造项目能力。
- 所有项目都被标为“强推荐”。
- 资源合集、模板或示例集合被误判为生产级深读项目。
- 推荐理由只有判断，没有可复核证据。
