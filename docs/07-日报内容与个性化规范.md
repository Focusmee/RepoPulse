# 日报内容与个性化规范

## 日报目标

日报不是信息堆砌，而是帮助用户做学习决策。读完日报后，用户应该能决定：

- 今天优先看哪 1 到 3 个项目。
- 为什么它们适合自己。
- 从哪里开始阅读。
- 是否值得做成 demo、简历项目或产品想法。

## 日报结构

推荐 Markdown 结构：

```markdown
# RepoPulse 日报 - YYYY-MM-DD

## 今日结论

用 3 到 5 句话总结今天的技术趋势和最值得关注方向。

## 今日最值得深读

### 1. owner/name

- 一句话定位：
- 推荐等级：
- 项目类型：
- 综合分：
- 置信度：
- 事实来源：
- 为什么值得关注：
- 学习价值明细：
- 为什么值得学习：
- 推荐理由与证据：
- 为什么适合当前画像：
- 推荐阅读路径：
- 可转化项目想法：
- 风险：
- 链接：

## 上升很快，值得观察

## 可转化为项目灵感

## 今日不推荐但值得知道
```

## 单项目卡片字段

每个项目必须包含：

| 字段 | 要求 |
| --- | --- |
| 项目名 | 使用 `owner/name` |
| 一句话定位 | 不超过 40 个中文字 |
| 推荐等级 | 强推荐 / 值得观察 / 项目灵感 / 谨慎关注 |
| 项目类型 | 说明是应用/库、框架/平台、开发者工具、资源合集、模板或示例集合 |
| 综合分 | 展示综合分、学习价值、热度和画像匹配 |
| 置信度 | 展示 AI/规则分析置信度和原因 |
| 事实来源 | 展示 GitHub metadata、README、release、topics、license |
| 为什么值得关注 | 展示关键增长指标、社区信号和近期活跃，不堆全部数据 |
| 学习价值 | 展示分数、分项和 2 到 4 条理由 |
| 推荐理由与证据 | 每条理由必须拆成“判断”和“证据” |
| 为什么适合当前画像 | 必须关联语言、topic 或学习目标 |
| 阅读路径 | 2 到 5 步，可执行 |
| 项目想法 | 可选，但对求职和应用开发很有价值 |
| 风险 | 至少 1 条 |
| 链接 | GitHub URL |

## 语气规范

日报语气应：

- 直接。
- 有判断。
- 不夸张。
- 不营销。
- 不用空泛形容词堆砌。

避免：

- “这是一个非常优秀的项目”但不给理由。
- “值得学习”但不说明适合谁。
- “可以提升技术能力”这种泛泛表达。
- 长篇复制 README。

## 个性化画像

用户画像至少包含：

```yaml
profile_id: default
role: "后端开发者"
preferred_languages:
  - Java
  - TypeScript
interested_topics:
  - distributed-systems
  - llm
  - agent
learning_goals:
  - 找工作
  - 学架构
  - 做可展示项目
excluded_topics:
  - blockchain
daily_limit: 10
```

## 个性化规则

日报必须体现画像：

- 语言匹配：偏好语言项目获得加分。
- 主题匹配：关注 topic 获得加分。
- 目标匹配：求职、架构学习、应用开发分别影响推荐理由。
- 排除项：命中排除 topic 的项目降权或过滤。
- 新鲜度：近期已推荐过的项目降低重复出现概率。

## 不同学习目标的解释重点

| 学习目标 | 推荐理由应强调 |
| --- | --- |
| 找工作 | 技术栈匹配、可写进简历、工程完整度 |
| 学架构 | 模块边界、设计模式、可读性、规模 |
| 读源码 | 目录清晰度、核心路径、测试和 examples |
| 做应用 | 问题场景、可改造空间、部署难度 |
| 跟趋势 | 增长信号、生态位置、相似项目 |

## 分层推荐规则

### 今日最值得深读

进入条件：

- `learning_score` 高。
- `profile_match_score` 高。
- 证据充分。
- 风险可接受。
- 数量最多 3 个。
- 推荐等级为“强推荐”的项目必须位于全局 Top 3，且不是资源合集、模板或示例集合。

### 上升很快，值得观察

进入条件：

- `trend_score` 高。
- 但项目成熟度、文档或用户匹配度未完全确认。
- 热度高但证据或成熟度仍需继续观察。

### 可转化为项目灵感

进入条件：

- 实战迁移价值高。
- 能改造成 demo、工具、插件、模板或 SaaS 小功能。
- 明显资源合集、awesome list、示例集合、starter、template、boilerplate 项目默认进入该层。
- 不是资源/模板/示例类的项目，不能只因为有 `project_idea` 就进入该层；还需要高实战迁移分、画像相关性和明确证据。

### 谨慎关注

进入条件：

- 热度高但有明显风险。
- 文档不足。
- 项目过早期。
- license 或维护状态不明。

## 质量警告

日报生成后应运行质量检查。质量警告不一定代表日报失败，但必须在报告中显式展示，方便人工复核。

首版质量警告包括：

- 强推荐数量超过 3。
- “今日最值得深读”数量超过 3。
- Top 项目全部落在同一个分区。
- 推荐理由证据为空或过于空泛。
- 出现异常 token，例如 `iii`、`READM`、`undefined`、`null`、`NaN` 或乱码字符。
- 出现过度营销化表达，例如“完美匹配”“极完备”“一应俱全”“最强”“必看”。

报告中应展示：

```markdown
## 质量警告

- medium / suspicious_token：owner/name：分析文本中出现异常 token：READM
```

## 日报长度

首版建议：

- Top 深读项目：最多 3 个。
- 值得观察项目：3 到 5 个。
- 项目灵感：2 到 3 个。
- 总篇幅控制在 10 分钟内能读完。

## 归档命名

建议：

```text
reports/
  2026/
    2026-05-15-default.md
```

如果有多个画像：

```text
2026-05-15-java-backend.md
2026-05-15-ai-builder.md
```

## 日报质量底线

一份日报如果出现以下情况，应视为不合格：

- 只列项目，没有判断。
- 推荐理由和用户画像无关。
- 学习价值分没有解释。
- 阅读路径不可执行。
- 风险全部为空。
- 明显复制 README 而没有分析。
- AI 编造项目能力。
- 所有项目都被标为“强推荐”。
- 资源合集、模板或示例集合被误判为生产级深读项目。
- 推荐理由只有判断，没有可复核证据。
