# MVP 需求说明

## MVP 目标

RepoPulse v0.1 的目标是打通最小闭环：

> 每天收集候选 GitHub 项目，补充项目上下文，计算热度与学习价值，结合用户画像排序，生成一份可阅读的 Markdown 日报。

v0.1 不追求界面复杂、数据源丰富或推荐模型完美，只追求稳定、可解释、可复盘。

## 用户输入

首版至少需要一份本地或数据库中的用户画像配置：

```yaml
preferred_languages:
  - Java
  - TypeScript
interested_topics:
  - distributed-systems
  - llm
  - agent
learning_goals:
  - 找工作
  - 学习工程架构
  - 发现可做成应用的项目
excluded_topics:
  - blockchain
daily_limit: 10
```

## 核心输出

每日生成一份 Markdown 日报，至少包含：

- 日期与用户画像摘要。
- Top 项目列表。
- 每个项目的一句话定位。
- 热度指标。
- 学习价值评分。
- 推荐理由。
- 适合人群。
- 推荐阅读路径。
- 潜在风险。
- GitHub 链接。

## 功能需求

| ID | 需求 | 优先级 | 验收方式 |
| --- | --- | --- | --- |
| FR-01 | 支持每日任务入口 | P0 | 可以手动或定时触发一次完整日报生成 |
| FR-02 | 采集 GitHub Trending 候选项目 | P0 | 单次采集不少于 20 个候选项目 |
| FR-03 | 通过 GitHub Search 补充 topic / keyword 候选项目 | P0 | 能按用户画像拉取额外项目 |
| FR-04 | 获取仓库基础元数据 | P0 | owner、name、description、stars、forks、language、topics 可用 |
| FR-05 | 保存仓库每日快照 | P0 | 同一仓库同一天重复执行不会重复膨胀 |
| FR-06 | 计算 1 日和 7 日趋势指标 | P0 | 有历史数据时能计算增量；无历史数据时可降级 |
| FR-07 | 获取 README 文本 | P0 | 成功获取主流仓库 README；失败时记录原因 |
| FR-08 | 生成结构化 AI 分析 | P0 | 输出符合 `05-AI分析与提示词契约.md` 的 JSON |
| FR-09 | 计算学习价值分 | P0 | 每个分数都有理由，不只给数字 |
| FR-10 | 结合用户画像排序 | P0 | 不同画像下排序结果应有明显差异 |
| FR-11 | 生成 Markdown 日报 | P0 | 日报可直接阅读与归档 |
| FR-12 | 邮件投递 | P1 | 可在 Markdown 稳定后加入 |
| FR-13 | Web dashboard | P2 | MVP 不做，仅保留后续扩展接口 |

## 非功能需求

| ID | 需求 | 标准 |
| --- | --- | --- |
| NFR-01 | 可重复运行 | 同一天重复运行不会制造重复项目或重复快照 |
| NFR-02 | 可降级 | 单个仓库失败不影响整份日报 |
| NFR-03 | 可解释 | 每个推荐必须有 2 到 4 条推荐理由 |
| NFR-04 | 可审计 | 保留候选来源、评分明细、AI 输出原始 JSON |
| NFR-05 | 可控成本 | AI 分析只对通过初筛的项目运行 |
| NFR-06 | 可维护 | collector、scorer、AI、report renderer 模块边界清晰 |

## 首版流程

```text
加载用户画像
→ 采集 Trending 候选项目
→ 按 topic / keyword 补充候选项目
→ 去重与基础过滤
→ 获取仓库元数据
→ 保存每日快照
→ 计算趋势指标
→ 选择 Top 候选进入 AI 分析
→ 获取 README / Release 等上下文
→ 生成结构化分析
→ 计算学习价值分与个性化排序
→ 渲染 Markdown 日报
→ 保存日报
```

## MVP 砍掉项

为了最快可用，v0.1 明确不做：

- 账号体系。
- 复杂前端。
- 向量搜索。
- 多语言日报。
- 自动反馈学习。
- GitHub 之外的大量数据源。
- 周报、月报。
- 项目相似度图谱。
- 团队协作功能。

## v0.1 完成定义

v0.1 完成必须满足：

1. 能连续生成 3 天日报。
2. 每天候选项目不少于 50 个。
3. 每天最终推荐项目不少于 10 个。
4. 随机抽查 20 个项目，至少 80% 的摘要无明显事实错误。
5. 同一批候选项目在两个不同用户画像下，Top 10 至少有 4 个不同项目。
6. 所有推荐都包含明确学习理由和阅读路径。
