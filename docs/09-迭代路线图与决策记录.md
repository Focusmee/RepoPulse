# 迭代路线图与决策记录

## 路线图原则

RepoPulse 的迭代顺序应遵循：

1. 先可用，再好看。
2. 先稳定日报，再复杂控制台。
3. 先规则可解释，再个性化复杂化。
4. 先少量高质量数据源，再扩展多源。
5. 先手动复核质量，再自动优化推荐。

## 阶段 0：文档与方向冻结

目标：

- 明确产品定位。
- 明确 MVP。
- 明确评分模型。
- 明确数据和 AI 契约。

交付：

- `docs/` 文档体系。
- 初始用户画像样例。
- 初始日报模板。

完成标准：

- 新功能能对应到文档。
- 不再把 RepoPulse 做成单纯热榜。

## 阶段 1：本地 MVP

目标：

- 本地运行每日任务。
- 生成 Markdown 日报。

范围：

- GitHub Trending。
- GitHub Search。
- README 获取。
- 基础快照。
- AI 结构化分析。
- 评分排序。
- Markdown 报告。

不做：

- Web dashboard。
- 多用户。
- 向量检索。
- 自动反馈学习。

完成标准：

- 连续 3 天可生成有效日报。
- 人工抽样质量达标。

## 阶段 2：稳定运行与归档

目标：

- 定时运行。
- 保存历史日报。
- 基础错误重试和日志。

范围：

- GitHub Actions 或 cron。
- Supabase / PostgreSQL 存储。
- 报告归档。
- 邮件投递。
- 基础观测指标。

完成标准：

- 连续 7 天任务成功率 >= 95%。
- 每天报告可追溯输入和评分依据。

## 阶段 3：产品化阅读体验

目标：

- 从“能生成日报”升级为“更好读、更好复盘”。

范围：

- 简单 Web 页面。
- 项目详情页。
- 历史日报检索。
- 用户画像配置页面。
- 周报。

不急着做：

- 复杂图谱。
- 团队协作。
- 大规模多用户。

完成标准：

- 用户能在 Web 中查看历史推荐和评分原因。

## 阶段 4：趋势与智能增强

目标：

- 从每日推荐扩展到长期趋势观察。

范围：

- 30 日趋势。
- 相似项目聚类。
- 向量检索。
- 项目对比。
- 反馈学习。

完成标准：

- 能回答“某个技术方向最近有哪些项目在上升”。
- 能回答“这个项目和过去哪些项目相似”。

## Backlog 取舍规则

新增需求进入 backlog 前，必须回答：

1. 是否帮助用户判断“值不值得学”？
2. 是否提高日报质量或稳定性？
3. 是否能被当前架构自然支持？
4. 是否会明显增加维护成本？
5. 是否属于当前阶段？

如果 1 和 2 都是否，默认不做。

重大决策必须追加 ADR。模板：

```markdown

```

## 当前 ADR

### ADR-001：首版以 Markdown 日报为主，不先做 Web dashboard

日期：2026-05-15

#### 背景

RepoPulse 的核心风险不是界面，而是推荐质量、评分解释和日报稳定性。如果一开始投入复杂 Web dashboard，容易推迟核心闭环。

#### 决策

v0.1 先生成 Markdown 日报。邮件和 Web 都建立在同一份结构化日报数据之上。

#### 备选方案

- 先做 Web dashboard：展示更直观，但开发成本高，且无法验证评分质量。
- 先做 CLI：实现快，但对长期阅读和归档不如 Markdown 直观。

#### 影响

- 报告模板和结构化项目列表必须设计好。
- 后续 Web 只消费已有日报数据，不重新发明推荐逻辑。

#### 回滚条件

如果 Markdown 日报无法支持有效阅读和复盘，再提前建设 Web 页面。

### ADR-002：学习价值分优先级高于热度分

日期：2026-05-15

#### 背景

RepoPulse 的差异化在“值得学习”，不是“最火”。如果最终排序过度依赖 star 增长，会退化成普通热榜。

#### 决策

最终 `personalized_score` 中 `learning_score` 权重初始高于 `trend_score`。

#### 备选方案

- 热度分主导：更容易捕捉爆款，但学习价值弱。
- AI 主观分主导：解释性强，但不稳定，且容易幻觉。

#### 影响

- 需要更认真设计学习价值证据。
- 中等热度但高质量项目有机会进入日报。

#### 回滚条件

如果日报长期错过重要上升项目，可提高 `trend_score` 权重或增加“上升很快，值得观察”分区。

### ADR-003：可信日报使用确定性分层后处理和质量警告

日期：2026-05-18

#### 背景

RepoPulse 接入真实模型分析后，日报能够稳定生成，但出现了两个可信度问题：

- Top 项目容易全部被标为“强推荐”或“今日最值得深读”，缺少编辑克制。
- awesome list、模板、示例集合等资源型项目可能因为学习价值分高而进入深读区。
- 模型输出偶尔会出现异常 token、拼写错误或偏营销化表达，需要显式暴露给用户复核。

如果只依赖 prompt 约束，模型风格变化仍可能破坏日报分层和推荐语气。

#### 决策

在排序完成后增加确定性后处理：

- “强推荐”最多 3 个，且必须位于全局 Top 3、学习价值高、置信度高、无 high risk。
- “今日最值得深读”最多 3 个。
- `awesome`、资源合集、示例集合、starter、template、boilerplate 类项目默认进入“可转化为项目灵感”。
- 仓库类型识别只使用仓库事实字段和项目摘要，不使用 `project_idea`。
- 非资源/模板/示例类项目只有在实战迁移价值高、画像相关且证据充分时，才进入“可转化为项目灵感”。
- 日报生成后运行质量检查，并在报告中展示质量警告。

#### 备选方案

- 只修改 prompt：实现成本低，但不能保证不同模型或不同批次输出稳定。
- 只按分数阈值分层：简单，但无法识别资源合集、模板和示例集合。
- 人工复核后再发布：质量最高，但无法满足每日自动化日报目标。

#### 影响

- 日报语气更克制，强推荐数量可控。
- 资源型项目仍可被推荐，但默认作为灵感而不是深读工程。
- 报告中会出现质量警告区，帮助用户判断哪些内容需要人工复核。
- 排序解释需要同步展示项目类型、事实来源、证据和风险。

#### 回滚条件

如果后处理导致高质量项目长期被压到低优先级，或质量警告噪声过高影响阅读，可调整分层阈值、项目类型识别规则和质量检查规则。

### ADR-004：AI 失败必须可诊断、可恢复

日期：2026-05-18

#### 背景

真实模型偶尔会返回坏 JSON、截断 JSON、schema 不完整或 provider 错误。过去日志里只能看到 `Unterminated string in JSON` 之类的摘要，无法复盘原始模型输出、输入上下文和失败类型，也无法判断日报中哪些项目是 heuristic 降级结果。

#### 决策

为 AI 调用增加可诊断和可恢复机制：

- `REPOPULSE_DEBUG_AI=1` 时，把失败上下文写入 `REPOPULSE_DEBUG_DIR`。
- debug artifact 保存原始响应、`choices[0].message.content`、输入摘要、错误类型和错误栈。
- JSON parse 失败后，用更短 README / release 输入和更强 JSON 约束重试一次。
- 重试仍失败时降级为本地 heuristic，不阻塞日报生成。
- 统计 AI 失败类型：`parse_error`、`schema_error`、`http_error`、`timeout`、`rate_limit`。
- 日报运行概览展示 AI 成功数、heuristic 降级数、失败类型分布。
- 项目条目展示是否为 heuristic 降级，以及对应失败类型和 debug 文件路径。

#### 备选方案

- 只在 console 输出原始响应：实现简单，但长响应难检索，也不利于后续复盘。
- 坏 JSON 直接失败整份日报：暴露问题更明显，但降低日报可用性。
- 自动修复 JSON 而不重试模型：能处理部分格式问题，但容易掩盖模型 schema 质量问题。

#### 影响

- 单个模型坏 JSON 不再影响整份日报生成。
- 调试 OpenAI-compatible provider 时可以直接定位是模型输出、HTTP 错误、限流还是 schema 问题。
- 报告读者可以知道哪些项目是 AI 分析，哪些项目是 heuristic 降级。
- debug artifact 默认写入 `data/debug-ai/`，该目录必须被 git 忽略，避免把模型原始响应、输入摘要或错误栈误提交。
- parse retry 可能产生多份 debug artifact，日报需要展示所有 attempt 的路径，避免最终失败类型和可查看文件不一致。
- AI 分析缓存必须带版本边界；当失败诊断字段或 prompt 契约变化时，应重新分析，避免复用旧的不可诊断降级结果。

#### 回滚条件

如果 debug artifact 体积过大或泄露不该保存的输入信息，可以进一步裁剪 `input_summary` 和响应长度；如果 parse retry 明显增加成本但收益低，可以改成只在小批量调试模式开启。

#### 后续风险

- `schema_error`、`http_error`、`timeout`、`rate_limit` 必须有回归测试覆盖，防止 provider 兼容性调整时失败分类退化。
- 如果未来支持更多 OpenAI-compatible provider，需要把 `response_format` 支持能力做成显式配置或探测，而不是只依赖文档提示。

### ADR-005：先做 HTML 精华版，PDF 通过打印样式或后续自动化导出

日期：2026-05-19

#### 背景

RepoPulse 开始验证面向 AI 应用开发者的付费日报订阅后，完整 Markdown 日报适合归档和深读，但不适合社交平台推广、私信试读和快速成交。推广材料需要一份更短、更像商品样例的版本，突出 Top 3 项目的判断、风险和行动建议。

同时，历史输出中已经存在 HTML/PDF 成品，但当前代码没有稳定的 HTML/PDF 生成脚本。如果直接引入自动 PDF 依赖，会增加维护成本，并提前优化尚未被验证的交付形态。

#### 决策

新增精华版 HTML 生成工具：

- 输入现有 Markdown 日报。
- 输出短版 HTML，保留 Top 3 深读项目和最多 2 个项目灵感。
- HTML 内置移动端和打印样式，可通过浏览器打印导出 PDF。
- 首版不引入 Puppeteer/Playwright 自动 PDF。
- 完整 Markdown 日报仍是核心数据源，精华版只作为推广、试读和成交包装层。

#### 备选方案

- 直接自动生成 PDF：便于发送附件，但会引入浏览器运行时依赖，且早期用户未证明 PDF 是必要交付形态。
- 只手工剪裁报告：实现最快，但难以稳定复用，也不利于多画像、多平台推广。
- 直接做 Web 页面：展示更完整，但超出 M0/M1 的变现验证范围。

#### 影响

- 推广材料可以快速从已有日报生成，不重跑采集和 AI 分析。
- 小红书、抖音、社群和私信用例可以统一使用 HTML 精华版截图或链接。
- PDF 需求先通过浏览器打印满足；若 M1 阶段出现批量 PDF 投递需求，再新增 `--format pdf`。

#### 回滚条件

如果用户更偏好完整 Markdown 或邮件正文，而精华版 HTML 没有提升试读转化率，可以暂停维护精华版模板，只保留抽取逻辑供邮件摘要复用。

#### 后续重构方向

当前精华版工具从 Markdown 反向抽取内容，适合作为 M0/M1 的过渡方案。后续应重构为两层：

- 报告模型构建层：从 `ranked.items`、`stats`、`profile` 和 `date` 生成完整报告模型，再提炼 full / brief / email / social card 等变体模型。
- 多格式渲染层：从报告模型渲染 Markdown、HTML、Email HTML 和 PDF，不负责业务判断。

长期不应让 HTML/PDF 渲染器依赖 Markdown 文本结构。

### ADR-006：报告展示中文化和项目背景解释优先于源码全量分析

日期：2026-05-19

#### 背景

RepoPulse 当前基础日报基于 GitHub 元数据、README 和 latest release 判断，不读取完整仓库源码。对 AI 应用开发者来说，这已经能支持“是否值得点开”的快速筛选；但报告里仍有 `low`、`medium`、`high` 等英文枚举，并且部分项目介绍对不熟悉技术背景的读者不够友好。

如果为了提高理解力直接读取全量源码，会增加 GitHub API 成本、模型成本、时延和失败面，也会让日报生成复杂度明显上升。

#### 决策

短期优先做展示层和解释层优化：

- 内部 schema 继续使用 `low` / `medium` / `high`，但报告展示统一中文化为低/中/高、低风险/中风险/高风险。
- Top 项目增加“项目背景与使用场景”“简单例子”“能学习到什么”，帮助不熟悉技术的读者快速理解。
- prompt 要求这些解释基于 README、release 和元数据，不得假装已经读取源码。

源码分析作为后续增强：

- 只对 Top 1-3 项目做目录树和关键文件轻扫。
- 不做全仓库 clone 和全量源码输入。
- 作为深度版或付费增强卖点，不作为基础日报前置条件。

#### 备选方案

- 立即全量读取源码：判断更深，但成本和复杂度过高，不适合快速付费验证。
- 只保留当前报告：实现成本最低，但新用户理解成本偏高，影响试读转化。
- 只靠精华版缩短内容：能减少阅读压力，但无法解决“不知道项目做什么、怎么用”的理解问题。

#### 影响

- 报告会更适合小红书、抖音、社群和私信试读场景。
- 内部排序和 schema 稳定性不受中文化影响。
- 后续如果引入源码轻扫，需要在数据源、AI 输入、缓存版本和报告置信度中明确标注。

#### 回滚条件

如果背景解释导致报告过长、重复 README 或降低专业读者阅读效率，可以只在精华版和 Top 3 中保留，完整版其他项目继续使用短字段。
