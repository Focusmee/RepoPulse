# 验收标准与测试计划

## 验收目标

RepoPulse 的验收重点不是“程序能跑”，而是“日报能帮用户做学习决策”。验收分为功能、数据、评分、AI、个性化、稳定性和内容质量。

## 功能验收

| ID | 项目 | 标准 |
| --- | --- | --- |
| AC-F01 | 完整任务 | 能从用户画像生成一份 Markdown 日报 |
| AC-F02 | 候选采集 | 单日候选项目不少于 50 个 |
| AC-F03 | 去重 | 同一仓库不会重复出现在日报 |
| AC-F04 | 快照 | 仓库每日数据可保存并重复运行不膨胀 |
| AC-F05 | 趋势计算 | 有历史数据时能计算 1 日、7 日增量 |
| AC-F06 | AI 分析 | 输出 JSON 可解析并通过 schema 校验 |
| AC-F07 | 报告生成 | 报告包含所有必填字段 |
| AC-F08 | 精华版 HTML | 能从现有 Markdown 日报生成适合试读和推广的 HTML 精华版 |

## 精华版验收

精华版用于社交平台推广、私信试读和付费转化。验收标准：

- 输入一份现有 Markdown 日报，能生成 HTML 文件。
- HTML 至少包含封面、今日结论、运行概览核心指标、Top 3 深读项目和 CTA。
- Top 3 项目必须包含项目链接、是否值得点开、推荐理由、最大风险和推荐动作。
- Top 3 项目应包含项目背景与使用场景、简单例子或等价解释，能让不熟悉该技术的读者快速理解项目做什么。
- 展示层不得直接暴露 `low` / `medium` / `high` 作为主要用户文案；应显示低/中/高、低风险/中风险/高风险等中文描述。
- GitHub 链接可打开，文本内容经过 HTML 转义。
- 缺少项目想法、风险或学习成本时仍能生成，只省略对应区块。
- 移动端宽度不溢出，浏览器打印预览可导出 PDF。
- 读者 5 分钟内能读完，30 秒内能判断 Top 3 是否值得点开。

## 数据准确性验收

抽样 20 个项目检查：

- owner/name 正确。
- GitHub URL 正确。
- star、fork、issue 数和采集时间一致。
- README 文本对应正确仓库。
- 来源标签准确。
- 7 日增量计算符合快照。

通过标准：

- 95% 以上基础元数据无明显错误。
- 100% 的日报链接可打开。
- 0 个重复仓库进入最终 Top 列表。

## 评分验收

准备三类样例：

1. 高热度但学习价值低。
2. 中等热度但工程和文档优秀。
3. 用户画像强相关但热度一般。

验收标准：

- 高热度低学习价值项目不会稳定占据 Top。
- 中等热度高学习价值项目有机会被推荐。
- 用户强相关项目能获得个性化加权。
- 每个推荐分数都有理由。
- 学习价值分和热度分不能完全同向。

## AI 质量验收

人工抽样至少 20 个 AI 分析结果，检查：

| 指标 | 通过标准 |
| --- | --- |
| 摘要准确性 | 80% 以上无明显事实错误 |
| 推荐理由 | 70% 以上有明确证据 |
| 阅读路径 | 70% 以上可直接执行 |
| 风险提示 | 70% 以上合理 |
| 画像关联 | 70% 以上能看出和用户目标有关 |
| 背景与场景解释 | 70% 以上能让不熟悉该技术的读者说清项目做什么、何时使用 |

严重问题：

- 幻觉项目功能。
- 编造 release、examples 或架构。
- 对明显无文档项目给高置信强推荐。
- JSON 经常无法解析。

## 个性化验收

至少准备两个画像：

```text
画像 A：Java 后端 / 分布式系统 / 中间件 / 找工作
画像 B：AI 工具 / Agent / LLM 应用 / 做应用
```

同一批候选项目下：

- 两份 Top 10 至少有 4 个不同项目。
- 每份日报至少 60% 推荐体现画像偏好。
- 推荐理由中必须出现“为什么适合你”。
- 命中排除 topic 的项目不得进入强推荐。

## 稳定性验收

连续运行 7 天：

| 指标 | 标准 |
| --- | --- |
| 日报任务成功率 | >= 95% |
| 单日候选项目 | >= 50 |
| 最终可排序项目 | >= 20 |
| README 获取成功率 | >= 70% |
| AI 分析成功率 | >= 80% |
| 单项目失败影响 | 不影响整份日报生成 |

## 回归测试清单

每次改采集、评分、AI 或日报模板后检查：

- 还能生成完整日报。
- schema 没有破坏。
- Top 项目不重复。
- 分数在 0 到 100。
- reasons 不为空。
- evidence 不为空。
- reading_path 不为空。
- 风险不为空。
- 用户画像仍能影响排序。
- 用户可见报告中的等级和风险已经中文化。
- 如果未启用源码轻扫，报告不得声称已阅读或验证源码实现。

## 人工复核流程

每周抽查一次：

1. 随机选 10 个推荐项目。
2. 打开 GitHub 页面和 README。
3. 对照日报摘要和推荐理由。
4. 标记准确、可疑、错误。
5. 将错误归类为采集问题、AI 问题、评分问题或日报模板问题。
6. 高优先级问题进入下一轮修复。

## 发布门槛

v0.1 发布前必须满足：

- 连续 3 天生成有效日报。
- 抽样 20 个项目，AI 摘要准确率 >= 80%。
- 两个画像的 Top 10 差异明显。
- 没有阻塞性数据错误。
- 文档与实现没有明显冲突。

## 失败复盘模板

```markdown
## 问题

发生了什么？

## 影响

影响了哪些日报、项目或用户画像？

## 原因

属于采集、数据、评分、AI、渲染还是调度问题？

## 修复

短期如何修？

## 预防

需要新增什么测试、文档约束或告警？
```
